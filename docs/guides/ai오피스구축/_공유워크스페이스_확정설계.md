## 0. 한 줄 요약
워크스페이스 = **사람 묶음**(개인/팀), 브랜드(시목 등) = 그 안의 **project**, 공유는 **항목별 `is_shared`**.
개인 공간도 **personal 워크스페이스 행**으로 통일 → 17개 테이블이 **단일 RLS 패턴**. 통합 뷰는 "필터 없는 쿼리".

---

## 1. 확정 모델 — 4계층 (역할 분리)

```
workspaces        = 사람 묶음 (권한 경계)         "누가 보나"   type: personal | team
   │  └ workspace_members (owner/member)
   ▼
projects          = 사업/활동 단위                "무슨 일이냐"  workspace_id FK
   │                 (시목·운명랩·쏠닝포인트, "이사 준비" 같은 비브랜드도 가능)
   ▼
brand_contexts    = AI 파이프라인 설정 (1:1)       "AI가 뭘 아냐"  project_id FK (브랜드 project만)
   ▼
items (tasks 등)  = 콘텐츠   workspace_id(NOT NULL) + is_shared + user_id + assignee_id
```

**구체 매핑 (확정)**
- **시목 = project.** 부부 **team 워크스페이스 "우리"**(멤버 {Sol, 남편}) 안에 project 시목.
- **운명랩·쏠닝포인트 = project**, Sol의 **personal 워크스페이스** 안.
- "운명랩을 나중에 공유" = `UPDATE projects SET workspace_id = 팀ws` 한 줄 (+ 하위 항목 이동).
- **개인 공간 = `type='personal'` 워크스페이스 행** (가입 시 자동 생성). `workspace_id` 는 **모든 콘텐츠에서 NOT NULL** (NULL 특례 없음).

## 2. 핵심 결정 (확정 표)

| 질문 | 확정 | 근거(fable) |
|---|---|---|
| 시목은 workspace? project? | **project** | 워크스페이스(사람) ≠ 브랜드(사업) — 다른 축 |
| projects가 공유단위 겸함? | **아니오** | 공유는 항목별 is_shared. 2중 규칙 금지 |
| brand_contexts 위치 | **별도 테이블, project_id FK** | 비브랜드 project엔 NULL 무더기 방지 |
| 개인 공간 | **personal 워크스페이스 행** | NULL이면 17테이블에 특례 분기 복제 → 누수 위험 |
| 통합 뷰 | **RLS 기반 "필터 없는 쿼리"** | 특별 기능 아님. VIEW엔 `security_invoker=true` 필수 |
| 기본 가시성 | **팀ws=공유 default / 일기(journals)=비공개 default** | "넣은 위치가 공유를 정함, is_shared는 예외 숨김" |
| 할일 추천 | **즉시 상대 할일 생성**(assignee=상대) + 푸시 | 사장님 결정. assignee는 항상 본인 할일 보이게 RLS 절 |
| 최우선 안전장치 | **AI 컨텍스트 스코핑** (§6) | service-role 경로가 RLS 우회 → 초대 전 필수 |

## 3. 데이터 설계 (확정 SQL 골격)

```sql
-- 3.1 신규
create table workspaces (
  id uuid primary key default gen_random_uuid(),
  name text not null, emoji text, color text,
  type text not null check (type in ('personal','team')) default 'team',
  invite_code text unique,                       -- 팀 초대용
  created_by uuid not null references auth.users(id),
  created_at timestamptz default now()
);
create table workspace_members (
  workspace_id uuid references workspaces(id) on delete cascade,
  user_id uuid references auth.users(id) on delete cascade,
  role text not null check (role in ('owner','member')) default 'member',
  nickname text, joined_at timestamptz default now(),
  primary key (workspace_id, user_id)
);
-- (선택) 협업 알림/피드 — 가볍게
create table workspace_activities (
  id uuid primary key default gen_random_uuid(),
  workspace_id uuid references workspaces(id) on delete cascade,
  actor_id uuid, action text, resource_type text, resource_id uuid,
  metadata jsonb, created_at timestamptz default now()
);

-- 3.2 projects 확장 (brand_contexts는 이전 AI오피스 검수안 유지: project_id unique FK)
alter table projects add column workspace_id uuid references workspaces(id);

-- 3.3 콘텐츠 테이블 공통 (tasks 예시 — schedules/insights/journals/readings/study_notes 동일)
alter table tasks
  add column workspace_id uuid references workspaces(id),   -- 백필 후 NOT NULL
  add column is_shared boolean not null default true,       -- ※ journals만 default false
  add column assignee_id uuid references auth.users(id);

-- 3.4 AI 파이프라인 산출물: project_id 보유, 가시성은 그 project의 workspace를 따름
--     daily_reports / agent_runs / detail_page_drafts 에 project_id (+ workspace_id 파생)
```

**타입별 default 가시성**
| 테이블 | is_shared default |
|---|---|
| tasks / schedules / insights / readings / study_notes / daily_reports | `true` (팀ws에 넣으면 공유 의사) |
| journals(아침·저녁·주간 일기) | **`false`** (프라이버시) |
> personal 워크스페이스에선 is_shared 무의미 → UI에서 토글 숨김.

## 4. RLS (확정) — 단일 패턴

```sql
-- 멤버십 함수 (STABLE, 쿼리당 1회 평가)
create or replace function my_workspace_ids()
returns setof uuid language sql stable
security definer set search_path = public as $$
  select workspace_id from workspace_members where user_id = auth.uid()
$$;

-- 모든 콘텐츠 테이블 공통 (tasks 예시)
create policy tasks_select on tasks for select using (
  user_id = auth.uid()                                       -- 내 것은 항상
  or assignee_id = auth.uid()                                -- 배정받은 것은 항상(추천 누수 방지)
  or (is_shared and workspace_id in (select my_workspace_ids()))
);
create policy tasks_insert on tasks for insert with check (
  user_id = auth.uid() and workspace_id in (select my_workspace_ids())
);
create policy tasks_update on tasks for update using (
  user_id = auth.uid() or assignee_id = auth.uid()
  or (is_shared and workspace_id in (select my_workspace_ids()))  -- 부부 공동 편집
);
create policy tasks_delete on tasks for delete using (user_id = auth.uid());
```
인덱스: `(workspace_id, is_shared)`, `(user_id)`, `(assignee_id)`.
**기존 17테이블 `user_id=auth.uid()` 정책은 안 지움** — RLS는 OR 합산이라 기존 동작 100% 보존.

## 5. 통합 뷰 — "필터의 부재" (security_invoker 필수)

```sql
create or replace view unified_agenda
with (security_invoker = true)        -- ★ 누락 시 RLS 우회 = 보안사고
as
select 'schedule' as kind, id, workspace_id, user_id, is_shared, title,
       date as on_date, time as on_time, null::text as status
from schedules
union all
select 'task', id, workspace_id, user_id, is_shared, title,
       due_date, null, status
from tasks where due_date is not null;
```
- **통합 화면** = `select * from unified_agenda where on_date between ...` (워크스페이스 필터 없음)
- **워크스페이스 화면** = 위 + `and workspace_id = :current`
- "정신없음" 해결은 **표현 계층**: 워크스페이스별 컬러 도트 + 다중선택 토글 칩.
- `Layout.tsx`에 `currentWorkspaceId`(null = 통합) 상태 → 모든 훅이 구독.

## 6. ⚠️ 최우선 안전장치 — AI 컨텍스트 스코핑 (남편 초대 전 필수)

**문제:** `context.ts`의 `buildSystemPrompt`는 유저의 *모든* 프로젝트·KPI·할일·요약을 주입.
Edge Function은 **service role → RLS 우회**. 남편이 시목 방에서 모디에게 물으면 Sol의 운명랩 KPI가 답변에 샐 수 있음.

**해결:** 컨텍스트 조합 쿼리를 **"현재 workspace + (is_shared OR 본인 소유)"** 로 직접 스코핑.
- `buildSystemPrompt(roomId, { workspaceId, userId })` 시그니처에 워크스페이스 추가.
- 데일리 파이프라인(`daily-office-run`)도 project→workspace 스코프로 데이터 수집.
- **이 작업을 초대(8단계)보다 먼저 끝내지 않으면 초대 순간이 누수 시작점.** (빌드 순서에서 9번)

## 7. UX (확정)

- **워크스페이스 스위처**(헤더): `[ 🌐 통합 ] / [ 👤 개인 ] / [ 🪵 우리(시목 팀) ]` — 다중선택 칩 + 컬러 도트.
- **작성/편집 모달:** 팀ws에선 `[✓] 공유` 기본 ON(일기만 OFF) + 할일은 `담당: (나/남편)`.
- **필터**(할일/기록/인사이트): `[전체] [내 것] [공유받은 것] [내가 추천한 것]`.
- **받은 할일:** `created_by ≠ assignee_id` 로 "받은 할일" 구분 표시 + 푸시(수락/거절 없이 즉시 내 할일).
- **AI 직원 같이 보기:** 시목 `daily_reports`(공유) → 부부 공통 대시보드 위젯. **푸시는 멤버 전원**(아래 8-주의).

## 8. 빌드 순서 (무중단, 가산적) — fable 9단계 채택

> 원칙: 모든 단계 additive / 기존 RLS 정책 유지 / **AI 스코핑(9)을 초대(8)보다 먼저**.

1. `workspaces`·`workspace_members`(+activities) 테이블 + RLS.
2. 기존 유저 전원 **personal 워크스페이스 + 멤버십 백필** + 신규가입 트리거(`auth.users` insert → personal ws 자동생성).
3. 콘텐츠 테이블에 `workspace_id`(nullable)·`is_shared`·(tasks)`assignee_id` 추가 — 앱 무영향.
4. 백필: 전 행 `workspace_id = 본인 personal ws`. 행 수 검증.
5. `NOT NULL` + FK + 인덱스 확정.
6. 새 SELECT/UPDATE 정책 추가(기존 정책 유지 = OR 합산).
7. 프론트 services에 workspace 컨텍스트(default=personal) → **스위처 출시 전까지 앱 동작 100% 동일** 확인.
   ★ CLAUDE.md 룰: `types.ts`/`*.service.ts`/`mockSupabase.ts` 3곳 동시 + **mockSupabase에 워크스페이스 필터링 시뮬**까지.
8. 스위처 UI + 시목 team ws 생성 + 남편 초대 + `projects.workspace_id` 이동(시목) + 시목 하위 기존 항목 이동(=**사용자 선택**).
9. **(8보다 먼저!)** Edge Functions(데일리 파이프라인·모디 컨텍스트) 워크스페이스 스코핑(§6).
   + `push.ts`가 단일 user 가정 → **멤버 전원 발송**으로 수정.
10. 이메일 초대 플로우(`workspace_members` pending invite) — 현재 스펙에 없으니 추가.

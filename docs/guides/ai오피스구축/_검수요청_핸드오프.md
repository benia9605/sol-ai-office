# A. 아키텍처 검수

## A1. 단일 cron `daily-office-run` 7스텝 순차 구조

**① 평가:** 발상은 맞지만, 이 구조 그대로는 프로덕션에서 거의 확실히 깨집니다.

**② 문제점**
- **타임아웃이 치명적.** Supabase Edge Function은 wall-clock 제한(플랜에 따라 수십~수백 초)이 있는데, 7스텝 × 3브랜드 = 최소 21회 LLM 호출입니다. Perplexity 검색형 호출은 한 번에 30~90초가 흔하고, 12섹션 상세기획 생성은 출력만 6~8K 토큰이라 단일 호출로 1~2분 걸립니다. 합산하면 10분 이상 — 단일 함수로는 불가능합니다.
- **부분 실패 = 전체 실패.** STEP4에서 API 에러가 나면 1~3의 결과(이미 토큰 소모)가 버려지고, 재실행하면 처음부터 다시 돕니다. `agent_runs`에 로그를 남기겠다고 했지만, 로그만 있고 **재개(resume) 메커니즘이 없습니다.**
- 3개 브랜드를 직렬로 도는 것도 낭비입니다. 브랜드 간엔 의존성이 전혀 없습니다.

**③ 개선안: "큐 테이블 + 셀프 리스케줄링 워커" 패턴**

`agent_runs`를 단순 로그가 아니라 **잡 큐로 승격**시키는 게 핵심입니다. 새 인프라(n8n, 외부 큐) 없이 기존 cron 패턴 그대로 갑니다.

```
[cron 05:00 · daily-office-seed]
  → 활성 브랜드별로 agent_runs에 (date, brand_id, step, status='pending') 행들 생성

[cron */3분 · daily-office-worker]  (05:00~07:00 시간창만 동작)
  → 의존성 충족된 pending 잡 1~2개 집어 status='running'으로 잠금
  → 해당 스텝 LLM 호출 → output을 agent_runs.output(JSONB)에 저장 → 'done'
  → 실패 시 'failed' + attempt 증가, attempt < 3이면 다시 'pending'
  → 모든 스텝 done이면 STEP7이 daily_reports로 조립 + 푸시
```

이 구조의 장점: 함수 1회 실행은 LLM 1~2콜이라 타임아웃 안전, 스텝 단위 재시도가 공짜로 생기고, 브랜드별 병렬도 자연스럽습니다. n8n류는 1인 운영에 관리 표면적만 늘리니 현 단계에선 비추천이고, 나중에 Supabase Queues(pgmq)로 갈아타는 건 쉬운 마이그레이션입니다.

## A2. 신규 DB 5개 테이블

**① 평가:** 테이블 선정 자체는 적절합니다. 컬럼 설계와 제약이 부족합니다.

**② 문제점 / ③ 개선안 (테이블별)**

- **`brand_contexts`** — 이미 `projects` 테이블이 있고 플랜에서 "브랜드=프로젝트 매핑"이라 했으면서 `brand_id`를 따로 두는 건 이중 정체성입니다. → `project_id` FK로 연결하세요. 추가 권장 컬럼: `compliance_level`('general' | 'health_food' — 쏠닝포인트 분기용), `quality_refs`(이 브랜드에 주입할 기준서 목록), `prohibited_words text[]`.
- **`daily_reports`** — `competitor, market, customer...`가 개별 TEXT 컬럼이면 나중에 구조 바꿀 때마다 마이그레이션입니다. → 스텝 산출물은 `agent_runs.output`에 두고, 이 테이블은 `(date, brand_id) UNIQUE` + `report(JSONB)` + `status('partial'|'complete')` + `token_total`로 단순화. **부분 실패 시에도 'partial' 리포트를 발행**할 수 있게 됩니다.
- **`agent_runs`** — A1대로 큐로 승격: `id, date, brand_id, step, status, attempt, output JSONB, tokens_in, tokens_out, cost_usd, error, started_at, finished_at` + `(date, brand_id, step) UNIQUE`.
- **`competitor_sources`** — `source_type('url'|'text')`, `is_active` 추가. URL은 어차피 자동 fetch가 막히는 경우가 많으니 `raw_text`가 주력이라는 걸 UX에 반영해야 합니다(아래 C9).
- **`detail_page_drafts`** — 테이블 자체는 OK인데, **이걸 데일리 파이프라인에 넣는 설계가 잘못됐습니다**(B6에서 상술). `sections JSONB`로 저장하면 기존 02 가이드의 "섹션별 재생성" UX와 바로 연결됩니다.
- **빠진 것:** 모든 신규 테이블에 `user_id` + RLS. Edge Function은 service role로 쓰더라도 프론트 조회는 RLS를 탑니다. 기존 17개 테이블과 동일 규칙 유지.

- **`context.ts`의 buildSystemPrompt는 재사용 불가**합니다 — 프론트(`src/services`) 코드라 Deno Edge Function에서 import할 수 없습니다. 현실적 해법은 Edge용 `_shared/brandPrompt.ts`를 별도로 작성하되, 브랜드 컨텍스트 조회 쿼리 로직만 동일하게 맞추는 것.
- **`extractActionItems` 재사용은 오히려 비추천.** AI 응답에서 사후 추출하는 건 채팅용 휴리스틱입니다. 배치에선 STEP7 프롬프트가 처음부터 `{ report_md, tasks: [{title, priority, due}] }` **구조화 JSON을 출력**하게 하는 게 훨씬 안정적입니다. 추출 단계 자체를 없애세요.
- mockSupabase 3곳 수정 룰은 신규 테이블의 **프론트 조회 화면**에만 해당(Edge Function은 무관). 리포트 뷰 페이지 만들 때 적용하면 됩니다.

# B. AI 로직·프롬프트 검수

## B4. 스텝별 모델 적합성 + 프롬프트 초안

| STEP | 플랜의 모델 | 검수 의견 |
|---|---|---|
| 1 경쟁사 | Perplexity | **둘로 쪼개야 함.** 1a) 사용자가 붙여둔 `competitor_sources.raw_text` 분석 = 검색 불필요 → **Claude Sonnet**. 1b) 신규 동향 검색 → Perplexity. 저장된 텍스트 분석에 검색 모델을 쓰는 건 비용·품질 모두 손해입니다. |
| 2 시장 | Perplexity | 적절. 단 매일은 과함(B6). |
| 3 고객 | Claude Sonnet | 모델은 맞는데 **입력 데이터가 없다는 게 진짜 문제**(D12). |
| 4 소구점 | GPT-4o | 품질상 문제는 없지만, 배치 파이프라인에 프로바이더 2개를 섞으면 에러 처리·재시도·비용 추적이 2벌이 됩니다. "마키 정체성"은 채팅 UX의 가치지 배치의 가치가 아닙니다. → **Claude Sonnet으로 통일 권장**(채팅방 마키는 GPT 유지). |
| 5 상세기획 | Claude **Opus** | **반대.** 역설계로 출력 골격(12섹션×6블록)이 완전히 고정된 상태에선 Sonnet이 충분히 따라옵니다. Opus는 토큰 단가가 수 배인데 매일 3브랜드 × 8K 출력이면 이 스텝 하나가 전체 비용의 70%를 먹습니다. |
| 6 인사이트 | Claude Sonnet | 적절. |
| 7 리포트 조립 | — | 종합·요약·tasks JSON 생성이라 **Haiku로 충분**. |

**프롬프트 초안 — 핵심 2개를 풀버전으로, 나머지는 골격만 드립니다.** (전체 7개 풀버전이 필요하면 다음 턴에 파일로 만들어 드릴게요.)

**STEP 1a. 경쟁사 소스 분석 (Claude Sonnet)**
```
당신은 {brand_name}의 경쟁사 분석 담당자입니다.

[브랜드 컨텍스트]
{brand_contexts: 카테고리/채널/톤/타깃/금지표현}

[어제까지의 분석 요약]
{전일 daily_reports.competitor 요약 — 300자}

[오늘의 입력: 사용자가 저장한 경쟁사 자료]
{competitor_sources.raw_text 목록}

규칙:
1. 어제 요약과 비교해 "새로 달라진 것"만 보고한다. 변화가 없으면
   "유의미한 변화 없음"이라고 정직하게 쓴다. 억지로 채우지 않는다.
2. 입력 자료에 없는 가격·리뷰 수치를 추정해서 쓰지 않는다.
   근거 없는 항목은 "자료 없음"으로 표기한다.
3. 출력은 JSON: { "changes": [...], "price_position": "...",
   "review_keywords": [...], "threats": [...], "no_change": bool }
```

**STEP 7. 리포트 + 할일 (Haiku)**
```
아래 6개 스텝 산출물을 종합해 1인 사업가의 아침 브리핑을 만듭니다.

[입력] STEP1~6 JSON 산출물
[출력 JSON]
{
  "report_md": "## 오늘의 핵심 3줄\n... ## 브랜드별 요약\n...",
  "tasks": [ { "title": "...", "priority": "high|medium|low",
               "reason": "어느 스텝의 어떤 발견에서 나왔는지 1줄" } ]
}
규칙: tasks는 최대 3개. "검토해보세요" 같은 모호한 할일 금지 —
오늘 30분 안에 실행 가능한 동사로 시작. 새 정보가 없는 날은 tasks 0개 허용.
```

나머지 골격: **STEP2**(Perplexity — "{카테고리} 한국 시장, 최근 7일 내 뉴스·트렌드만, 출처 URL 필수, 추측 금지"), **STEP3**(Sonnet — STEP1·2 결과 + brand_contexts.target 입력, 페르소나·페인포인트·반론 출력, *"입력에 실고객 데이터가 없으므로 가설임을 명시"* 강제), **STEP4**(Sonnet — 07 기준서의 피처→베네핏 번역공식 + 12 컬러심리 요약본 주입, 소구점 3개 + 각 헤드라인 2안), **STEP6**(Sonnet — 1~4 종합, "이번 주 안에 행동으로 옮길 수 있는 것"만 인사이트로 인정).

## B5. 역설계 상세페이지봇 로직의 STEP5 이식

**① 평가:** 더미#1 분석 품질이 좋고, 추출한 골격(전략브리핑 6항목 → 12섹션 × 6블록 고정 → 마무리 가이드)은 그대로 시스템 프롬프트화 가능합니다.

**② 문제점**
1. 그 봇은 **대화형 UX**입니다 — "다음"/"마지막" 페이지네이션, "섹션 번호만" 수정. 배치에선 이 인터랙션이 전부 무의미하므로 제거해야 합니다.
2. 마크다운 통짜 출력이면 `detail_page_drafts.sections`에 넣어도 섹션별 재생성을 못 합니다.
3. **건기식 컴플라이언스 분기(더미#2)가 미검증** 상태인데, 쏠닝포인트가 바로 그 케이스입니다. 검증 전에 이식하면 가장 리스크 큰 브랜드가 가장 약한 로직을 탑니다.

**③ 개선안**
- 출력을 `{ strategy_brief: {...}, sections: [{ id, name, headline, body, bullets[7], visuals[3], doubt_removal, mini_cta }], visual_guide, upsell }` JSON으로 강제. 페이지네이션은 max_tokens 한도 대응용 "섹션 1~6 / 7~12" 2회 호출로 대체.
- 컴플라이언스는 봇 검증을 기다리지 말고 **자체 07 기준서의 룰을 직접 하드코딩**하세요(B7). 봇의 더미#2는 "우리 룰이 빠뜨린 게 있나" 확인용으로만.
- 불릿 7개·비주얼 3개 고정은 좋은데, JSON 스키마 검증을 코드에서 한 번 더 — LLM이 6개만 줄 때가 있습니다.

## B6. 7개 작업의 의존성·순서

**① 평가:** 순서 자체는 논리적입니다. 진짜 문제는 순서가 아니라 **주기**입니다.

**② 문제점 — 이게 이 기획 전체에서 가장 큰 구멍이라고 봅니다:**
- **STEP5(상세페이지 기획)를 매일 돌리는 건 무의미합니다.** 상세페이지는 상품 단위 산출물인데, 상품 정보가 어제와 같으면 매일 거의 동일한 12섹션이 재생성됩니다 = 토큰 낭비 + 리포트 노이즈. 게다가 전체 비용의 대부분이 이 스텝입니다.
- **경쟁사·시장도 매일 새 정보가 나오지 않습니다.** "diff 개념" 없이 매일 돌리면 LLM은 빈손으로 그럴듯한 리포트를 *지어냅니다*. 데일리 자동 리포트의 최대 함정은 환각이 아니라 **"매일 새로운 척하는 환각"**입니다.
- 1·2를 직렬로 둘 이유가 없습니다(상호 독립).

**③ 개선안 — 주기 분리:**
```
데일리 (05:00):  (1a∥1b∥2 병렬) → 3 → 4 → 6 → 7
                 단, 모든 스텝에 "전일 결과 요약 + 변화분만 보고" 주입
위클리 (월요일):  5 상세·디자인 기획 (브랜드당 대표상품 1개)
온디맨드:        5를 인풋폼에서 수동 트리거 (Phase 1의 수동버전이 그대로 이 역할)
```
"새 정보 없음"을 정직하게 보고하게 허용하는 것이 품질의 핵심입니다.

## B7. 컴플라이언스 하드코딩

프롬프트만 믿으면 안 됩니다. **3겹 방어**를 권합니다:

1. **프롬프트 레이어:** `brand_contexts.compliance_level = 'health_food'`일 때 시스템 프롬프트에 고정 블록 주입 — "질병의 예방·치료 효능 표현 절대 금지. 기능성은 식약처 고시 문구('~에 도움을 줄 수 있음')만, 입력에 '수면·피로' 등 효능 암시가 있어도 '잠들기 전 루틴' 같은 *상황 묘사*로 순화. '1위/최고'는 기준·날짜·출처 없이는 금지."
2. **정규식 1차 필터(코드, 무료):** 생성 결과에서 `치료|예방|완치|개선됩니다|혈당|불면증` 등 금지·위험 단어 스캔.
3. **Haiku lint 2차(저비용):** "다음 텍스트에서 식품표시광고법 위반 소지 문장을 찾아 JSON으로" → 발견 시 리포트에 ⚠️ 플래그. 자동 수정까진 하지 말고 사람 확인으로 넘기는 게 안전합니다.

# D. 우선순위·다음 단계

## D11. Phase 로드맵 재정렬

원안(Phase 0→1 수동봇 3개→2 자동화→3 대시보드→4 입력UX)의 문제: **Phase 4(경쟁사 입력 UX)가 맨 뒤인데, 입력 데이터 없이는 Phase 2의 STEP1·3이 환각 제조기가 됩니다.** 또 "가장 빨리 가치"는 수동 상세페이지봇이 아니라 — 그건 지금도 외부 GPTs로 뽑을 수 있으니 — **매일 아침 리포트의 최소 버전**입니다.

재정렬 제안:

1. **(0.5d) Phase 0 유지:** `brand_contexts` + 기준서 요약판 분리 + context.ts 브랜드 주입.
2. **(1d) 미니 파이프라인 먼저:** 기존 `morning-briefing`을 확장해 *STEP 1b(Perplexity 동향) + 6 + 7만* 도는 "데일리 브리핑 v2". 기존 검증된 함수의 확장이라 가장 빠르고, 자동 리포트의 체감 가치를 즉시 확인 — 여기서 "매일 받을 만한가?"를 검증한 뒤에 나머지를 짓는 게 순서입니다.
3. **(1d) 경쟁사 소스 입력 UX** (원래 Phase 4를 앞으로) + 네이버 API 연동.
4. **(1~2d) STEP 1a·3·4 추가** → 풀 데일리 파이프라인 완성 (A1 큐 구조로).
5. **(1~2d) 상세페이지봇 수동버전** (각 방 인풋폼) — 이게 곧 STEP5 엔진.
6. **(0.5d) STEP5 위클리 자동화 + 대시보드 위젯.**

## D12. 놓치고 있는 것

1. **"고객 분석"의 입력 데이터가 시스템에 없습니다.** 리뷰·문의·판매 수치 없이 STEP3은 brand_contexts.target을 다시 풀어쓰는 작문이 됩니다. 최소한 `competitor_sources`처럼 **자사 리뷰/문의 붙여넣기 테이블**(또는 스마트스토어 리뷰 수동 CSV 업로드)을 추가해야 STEP3·4가 땅에 발을 딛습니다. 이게 빠진 게 기획 전체에서 두 번째로 큰 구멍입니다.

# 💬 CS 매니저 — 전문 SOP

> **역할:** "문의 답변 + 후기 관리" · outputKind `ticket_list`
> **재료:** `03_AI_CS_AGENT.md` + 3사 검증(분류 Haiku·답변 Sonnet·분쟁 HITL) + CS 실무.
> **이 문서:** 분류 체계 + 답변 프레임 + 시목 예시. 핵심은 코드 `BASE_SOP.cs`에 압축 반영.
> ⚠️ **하단에 GPT·퍼플렉시티 검증 답변을 붙일 자리**를 비워뒀어요(소싱과 동일 방식).

---

## 1. 철학 — "불편을 먼저 끌어안고, 정책 안에서 최대한 해결."

CS는 응대가 아니라 **신뢰 관리**다. 고객이 화났을 때 변명하면 후기로 남고, 공감하면 단골이 된다. 빠르되 기계적이지 않게 — **첫 문장은 항상 사람의 공감**, 그 다음에 해결. 정책상 안 되는 것도 "안 됩니다"가 아니라 **대안**으로 말한다.

---

## 2. 분류 체계 — 3축 + 상태 (검증 반영)

모든 문의를 **유형 × 긴급도 × 감정**으로 분류하고, **처리 상태·승인 필요**를 함께 매긴다.

| 축 | 값 |
|---|---|
| **유형** | 배송 / 교환·환불 / 상품문의 / **후기**(공개·별도) / **주문변경** / 기타 |
| **긴급도** | 즉시(불만·오배송·파손) / 일반 / 낮음 |
| **감정** | 긍정 / 중립 / 불만 / **격앙** |
| **처리 상태** | 신규 / 답변초안 / 승인대기 / 완료 / 보류 |
| **승인 필요** | true / false |

- **이탈위험 신호**(법적 언급·환불 강경·경쟁사 언급·탈퇴/취소)가 있으면 감정과 별개로 위험 플래그.
- **후기는 별도 유형** — 문의는 1:1, 후기는 공개(다른 고객도 봄).
→ **불만 × 즉시**는 큐 최상단. **`교환·환불 × 즉시 × (불만 또는 이탈위험)` → 승인 필요 자동 true.**

---

## 3. 답변 프레임 — 공감 → 해결 → 마무리 (3~5문장)

1. **공감(첫 문장 필수)**: 고객 상황을 **구체적으로**. "기다리셨을 텐데 많이 불편하셨겠어요."
2. **사실확인 + 해결 또는 대안**: 가능하면 바로 해결, 불가하면 명확히 + 대안. **환불·교환은 정책 내 최대 수용적**, 단 **무리한 약속 금지**.
3. **마무리**: **처리 시간을 구체적으로**(예: 영업일 1~2일) + 추가 도움 안내.

> 톤은 회사 브레인을 따른다(시목=정중·상세·신뢰 / 운명랩=친근·이모지 O / 쏠닝포인트=명확·건강 관련 신중).

### 3-1. 금지 표현 → 대체 표현 사전 (GPT 검증 — CS 핵심)
| ❌ 금지 | ✅ 대체 |
|---|---|
| 정책상 안 됩니다 | 바로 도와드리긴 어렵지만, 가능한 방법을 안내드릴게요 |
| 고객님 과실입니다 | 사용 중 생길 수 있는 부분이라 먼저 상태를 확인해볼게요 |
| 상세페이지에 적혀 있습니다 | 안내가 눈에 잘 안 띄었을 수 있어 다시 정리해드릴게요 |
| 환불 불가합니다 | 현재 기준으론 환불이 어렵지만, 대신 ○○를 도와드릴 수 있어요 |
| 원래 그런 제품입니다 | 제품 특성상 나타날 수 있는데, 불편하셨을 테니 확인 도와드릴게요 |
| 저희는 책임 없습니다 / 확인해보세요 | (사용 금지 — 책임 회피·떠넘기기 인상) |

## 4. 부정 후기 · 민감 건 (HITL)

### 부정 후기는 "문의"와 다르게 답한다 (공개 응대 — 미래 고객이 본다)
> 공감 → **책임 회피 없이 유감** → 해결 의지 → 개선 약속 → **개별 연락 안내**.
> 고객 잘못 공개 지적 금지 / 정책 논쟁 공개 금지 / 개인정보·주문정보 공개 요청 금지.

### 자동 발송 금지 = 사람 승인(HITL) 필수 조건 (검증 반영 — 9종)
환불·교환 금액 큰 건 / 법적·신고·소비자원·내용증명 언급 / 욕설·협박·악성 클레임 / 제품 안전 문제 / **건강식품 효능·부작용 문의** / 개인정보 판단 / 배송사고 보상 판단 / 반복 불만 고객 / 공개 부정 후기 → **초안만, 발송 전 사람 승인**.
> MVP 단계는 **모든 답변을 초안→사장 복사/발송**(자동 발송 X). 추후 배송조회·영업일·확정 FAQ 정도만 자동 후보.

### 반복 문의 → 개선 루프 (CS의 진짜 가치)
FAQ로 지식화 + "설명 부족으로 생긴 문의"는 **상세페이지/상품/비주얼 직원에게 인사이트로 넘긴다**.
예: "관리법 5건 → 상세 상단 관리법 요약 / 사이즈 체감 3건 → 손과 함께 든 사이즈 컷".

## 5. 출력 구조 (ticket_list)

```
🔴 [불만·즉시] "도마에 곰팡이가…"
   → 답변 초안: "불편을 드려 정말 죄송합니다. 원목은 첫 사용 전 오일 관리가 필요한데
      안내가 부족했네요. 교환 도와드리고 관리법도 함께 보내드릴게요."
   유형 교환·환불 / 감정 불만 / [복사] [FAQ에 반영] [⚠️발송은 승인 후]
🟡 [배송문의·일반] → 답변 초안 (단순 건은 바로 복사)
[다음 액션] 반복 문의 "관리법" 3건 → 상세페이지에 '관리법 섹션' 보강 인사이트
```
> **저장은 JSON 병행**(MVP-2): `{ tickets:[{ type, urgency, sentiment, draft, needsConfirm, needsHumanApproval }] }` — UI 큐·자동 FAQ·승인 게이트용.

## 7. 시목 Few-shot 예시 — 원목 도마 곰팡이 문의

```
🔴 [교환·환불 · 즉시 · 불만]
고객: "산 지 2주 된 도마에 곰팡이 같은 게 생겼어요. 불량 아닌가요?"

답변 초안:
"불편을 드려 정말 죄송합니다. 통원목 도마는 첫 사용 전·후 오일 관리가 필요한데,
안내가 충분치 못했던 것 같아 마음이 무겁습니다. 우선 교환 도와드리고, 다시 생기지 않도록
관리법 카드도 함께 보내드릴게요. 사진 한 장만 주시면 바로 처리해 드리겠습니다."

확인필요: 곰팡이 vs 오일 얼룩 — 사진 확인 후 교환/안내 분기 (→ 발송 전 승인)
다음 액션: '원목 관리법' 문의 반복 → 상세페이지에 관리법 섹션 보강 인사이트
```

좋아요. CS 매니저 SOP도 방향은 좋습니다.
특히 “첫 문장은 공감”, “정책 안에서 대안 제시”, “민감 건은 HITL” 이 3개는 반드시 유지해야 합니다.

다만 CS 직원은 소싱기획자보다 더 조심해야 합니다.
왜냐하면 CS는 잘못 답하면 바로 환불 분쟁, 후기 악화, 브랜드 신뢰 하락으로 이어지기 때문입니다.

결론부터 말하면, 이 직원은 단순히 “답변 초안 생성 직원”이 아니라 아래 역할로 잡아야 합니다.

CS 매니저 = 문의 분류자 + 답변 초안 작성자 + 위험 감지자 + FAQ/상세페이지 개선 제안자

1. SOP 점검 — 좋은 점

1) 철학이 좋음

현재 문장 좋습니다.

“불편을 먼저 끌어안고, 정책 안에서 최대한 해결.”

CS 직원의 정체성으로 아주 적합합니다.
특히 “안 됩니다”가 아니라 “대안”으로 말한다는 방향이 좋습니다.

다만 BASE_SOP에는 조금 더 명확하게 넣는 게 좋습니다.

너는 고객을 이기려는 직원이 아니라, 브랜드 신뢰를 지키는 CS 매니저다.
첫 문장은 반드시 고객 감정에 대한 공감으로 시작한다.
정책상 불가능한 요청도 단정적으로 거절하지 말고, 가능한 대안을 먼저 제시한다.

⸻

2) 3축 분류 구조가 실무적임

현재 분류 체계:

유형 × 긴급도 × 감정

이 구조는 좋습니다.

다만 실무에서는 여기에 처리 상태와 승인 필요 여부가 추가되어야 합니다.

현재:

축	값
유형	배송 / 교환·환불 / 상품문의 / 기타
긴급도	즉시 / 일반 / 낮음
감정	긍정 / 중립 / 불만

추천 보완:

축	값
유형	배송 / 교환·환불 / 상품문의 / 후기 / 주문변경 / 기타
긴급도	즉시 / 일반 / 낮음
감정	긍정 / 중립 / 불만 / 격앙
처리 상태	신규 / 답변초안 / 승인대기 / 완료 / 보류
승인 필요	true / false

특히 후기는 별도 유형으로 빼는 게 좋습니다.
문의와 후기는 성격이 다릅니다.

문의는 1:1 대응이고, 후기는 다른 고객도 보는 공개 응대입니다.
따라서 후기 관리는 별도 분류가 필요합니다.

⸻

2. SOP 보완점

보완 1. “자동 발송 금지 조건”을 더 명확히 해야 함

현재 HITL 조건이 잘 들어가 있지만, 더 구체화해야 합니다.

CS 직원은 아래 조건이 있으면 절대 자동 발송하지 않고 초안만 작성해야 합니다.

- 환불/교환 금액이 큰 경우
- 법적 조치, 신고, 소비자원, 내용증명 언급
- 욕설/협박/악성 클레임
- 제품 안전 문제
- 건강식품 효능/부작용 관련 문의
- 개인정보 요청/수정
- 배송 사고로 보상 판단이 필요한 경우
- 반복 불만 고객
- 공개 부정 후기

특히 쏠닝포인트처럼 건강식품이 있으면, “먹고 탈이 났다”, “효과가 없다”, “부작용 같다” 같은 문의는 무조건 승인대기로 가야 합니다.

⸻

보완 2. 답변 프레임에 “금지 문장”이 필요함

CS는 좋은 문장보다 하면 안 되는 말이 더 중요합니다.

예를 들어 아래는 금지해야 합니다.

고객님 과실입니다.
정책상 안 됩니다.
저희는 책임 없습니다.
상세페이지에 적혀 있습니다.
원래 그런 제품입니다.
환불 불가합니다.
확인해보세요.

대신 이렇게 바꿔야 합니다.

나쁜 표현	대체 표현
정책상 안 됩니다	바로 도움드리긴 어렵지만, 가능한 방법을 안내드리겠습니다
고객님 과실입니다	사용 과정에서 생길 수 있는 부분이라 먼저 상태를 확인해보겠습니다
상세페이지에 적혀 있습니다	안내가 충분히 눈에 띄지 않았을 수 있어 다시 한번 정리해드리겠습니다
환불 불가합니다	현재 기준으로는 환불 처리가 어려우나, 대신 ○○ 방법을 도와드릴 수 있습니다
원래 그런 제품입니다	제품 특성상 나타날 수 있는 부분이지만, 불편하셨을 수 있어 확인 도와드리겠습니다

이 “금지→대체 표현” 사전은 CS 직원 SOP에 꼭 넣는 게 좋습니다.

⸻

보완 3. 문의 답변과 후기 답변을 분리해야 함

문의 답변은 조금 더 직접적이어도 됩니다.
하지만 공개 후기는 브랜드 이미지를 같이 관리해야 합니다.

문의 답변

공감 → 확인 요청 → 해결 안내 → 다음 절차

부정 후기 답변

공감 → 책임 회피 없이 유감 표현 → 해결 의지 → 개선 약속 → 개별 연락 안내

부정 후기는 고객 한 명에게만 답하는 게 아닙니다.
미래 고객이 “이 브랜드는 문제 생기면 어떻게 대응하지?”를 보는 자리입니다.

그래서 후기 답변에는 아래 원칙이 필요합니다.

- 고객 잘못을 공개적으로 지적하지 않는다.
- 정책 논쟁을 공개 댓글에서 길게 하지 않는다.
- 개인정보/주문정보를 공개적으로 요청하지 않는다.
- 해결 의지와 개선 태도를 보여준다.

⸻

보완 4. “반복 문의 → 상세페이지 개선” 연결이 아주 중요함

현재 문서에 이 내용이 들어간 건 좋습니다.
이건 CS 직원의 진짜 가치입니다.

단순히 문의에 답하는 직원이면 시간이 조금 줄어드는 정도입니다.
하지만 반복 문의를 찾아서 상세페이지/상품기획/비주얼 직원에게 넘기면 문의 자체가 줄어듭니다.

이 부분은 페이지에 꼭 보여줘야 합니다.

예:

이번 주 반복 문의
- 원목 도마 관리법 5건
- 배송일 문의 3건
- 사이즈 체감 문의 3건
추천 개선
- 상세페이지 상단에 관리법 요약 추가
- 상품 이미지에 손과 함께 든 사이즈 컷 추가
- 배송 안내 문구를 구매 버튼 근처로 이동

4. CS 직원 페이지 핵심 방향

CS 직원 페이지는 소싱기획자와 다르게 큐 중심이어야 합니다.

소싱기획자는 “분석 리포트”가 중심이고,
CS 매니저는 “처리해야 할 티켓 목록”이 중심입니다.

그래서 페이지에 들어갔을 때 사장이 가장 먼저 봐야 하는 건 이것입니다.

지금 급한 문의가 몇 개인가?
사람 승인이 필요한 답변이 무엇인가?
복사해서 바로 보낼 수 있는 답변은 무엇인가?
반복 문의로 개선해야 할 내용은 무엇인가?

18. CS 직원에서 특히 조심할 점

1) “자동 발송”은 MVP에서 하지 않는 게 좋음

초기에는 무조건:

AI 답변 생성 → 사장 복사/수정 → 직접 발송

이 구조가 안전합니다.

나중에 단순 배송 문의 정도만 자동 발송 후보로 열 수 있습니다.

자동 발송 가능 후보:

- 배송 조회 안내
- 영업일 안내
- 기본 상품 사이즈 안내
- 이미 확정된 FAQ 답변

자동 발송 금지:

- 환불/교환
- 부정 후기
- 법적 언급
- 건강식품 효능/부작용
- 고객 불만이 강한 문의

⸻

2) 브랜드별 톤 차이가 중요함

시목과 운명랩, 쏠닝포인트는 CS 톤이 달라야 합니다.

브랜드	톤
시목	정중, 차분, 상세, 신뢰
운명랩	따뜻, 친근, 감정 수용, 이모지 일부 가능
쏠닝포인트	명확, 신뢰, 과장 없는 안내, 건강 관련 조심

회사 브레인에 CS 톤 필드를 따로 두면 좋습니다.

cs_tone:
- 첫 문장 공감 강도
- 이모지 사용 여부
- 문장 길이
- 환불/교환 대응 태도
- 금지 표현

⸻

3) 정책 DB가 필요함

CS 답변은 회사 브레인만으로 부족합니다.
배송/교환/환불 정책이 별도 DB로 있어야 합니다.

추천 테이블:

cs_policies (
  id uuid primary key,
  workspace_id uuid,
  category text, -- 배송, 교환, 환불, 파손, 후기
  title text,
  policy_text text,
  customer_facing_text text,
  updated_at timestamptz
)

예:

파손 도착:
- 수령 후 7일 이내 사진 확인 시 교환 가능
- 포장 박스 사진 필요
- 고객 안내문은 부드럽게

정책 DB가 없으면 AI가 “좋은 말”은 쓰지만, 실제 운영 기준과 어긋날 수 있습니다.

⸻

19. 최종 판정

CS 매니저 SOP는 큰 방향 합격입니다.
다만 실무형으로 만들려면 아래를 꼭 추가하세요.

반드시 수정

후기를 문의와 별도 유형으로 분리
자동 발송 금지 조건 명확화
금지 표현/대체 표현 사전 추가
처리 상태값 추가
정책 DB 연결
반복 문의 → FAQ/상세페이지 개선 흐름 강화

페이지에서 반드시 보여줄 것

1. 즉시 처리 문의 수
2. 승인 필요 문의 수
3. 티켓 큐
4. 문의 원문
5. AI 분류 근거
6. 답변 초안
7. 확인 필요 사항
8. 사람 승인 여부
9. FAQ 후보
10. 상세페이지/상품 개선 액션

한 줄 결론

CS 매니저는 “답변 작성봇”이 아니라 “고객 불편을 신뢰 회복과 상품 개선으로 연결하는 직원”이어야 합니다.

페이지도 그래서 단순 채팅창이 아니라, 아래 구조가 되어야 합니다.

문의 큐 → 위험 분류 → 답변 초안 → 승인/복사 → FAQ화 → 상세페이지 개선

이 흐름이 잡히면 CS 직원은 단순히 시간을 줄이는 수준을 넘어서, 후기 악화를 막고 상품 설명을 계속 개선하는 직원이 됩니다.

CS 매니저 SOP도 방향성은 아주 좋고, 최신 CX/AI-CS 베스트프랙티스랑 잘 맞습니다. 여기에 **분류 축·HITL 규칙·FAQ/인사이트 연동**을 조금만 더 구조화하고, 직원 페이지 레이아웃을 “티켓 큐 중심”으로 짜면 바로 실전에 투입 가능한 수준입니다. [tommasomariaricci](https://www.tommasomariaricci.com/blog/ai-for-customer-service-guide-2026)

***

## 1. SOP 내용 점검

### 1-1. 철학·역할 정의

- “불편을 먼저 끌어안고, 정책 안에서 최대한 해결” + “첫 문장은 항상 공감”은 **요즘 AI CS 설계에서 강조하는 ‘공감→해결→마무리’ 프레임과 정확히 일치**합니다. [pedowitzgroup](https://www.pedowitzgroup.com/how-do-ai-agents-handle-emotional-customer-interactions-care-guide)
- “안 됩니다” 대신 대안 제시, 정책 내 최대 수용, 신뢰 관리 관점도 잘 잡혀 있어서, BASE_SOP 상단에 그대로 넣으면 좋겠습니다.  

→ 철학 레벨은 이미 매우 좋고, **첫 문장은 반드시 공감 문장으로 시작**하도록 BASE_SOP에 강하게 명시하면 됩니다. [forbes](https://www.forbes.com/sites/yolarobert1/2026/04/14/3-simple-ways-to-uplevel-your-agentic-ai-customer-service/)

***

### 1-2. 분류 체계(유형 × 긴급도 × 감정) 점검

LLM 기반 CS에서 “intent(유형) + urgency + sentiment” 3축으로 triage 하는 것이 거의 표준이라, 지금 축 설계는 정석에 가깝습니다. [thinkbot](https://thinkbot.agency/blog/ai-driven-customer-service-automation-with-n8n-auto-triage-tickets-update-crm-and-send-personalized-email-follow-ups)

- 유형: 배송 / 교환·환불 / 상품문의 / 기타  
- 긴급도: 즉시 / 일반 / 낮음  
- 감정: 긍정 / 중립 / 불만  

**보완하면 좋은 점**

1. **“감정”을 조금 더 세분하거나 churn 신호를 추가**  
   - 예: `긍정 / 중립 / 불만 / 분노` 혹은 `불만` 안에서 `불만(보통) / 이탈위험(법적 언급·환불 강경·경쟁사 언급)` 같은 식. [freeacademy](https://freeacademy.ai/lessons/cs-ticket-triage)
   - BASE_SOP에는 “법적 언급, 환불 강력 요구, 경쟁사 언급, 탈퇴/취소 언급”을 이탈·민감 신호로 명시.  
2. **HITL 조건을 축과 연결해서 더 명시적으로**  
   - 예:  
     - `유형=교환·환불 AND 긴급도=즉시 AND (감정=불만 OR 이탈위험)` → `needsHumanApproval=true`. [blog.eif](https://blog.eif.am/ai-for-customer-service-guide-2026/)

→ 지금 3축은 이미 좋은데, BASE_SOP에서 “이 조합이면 무조건 사람 승인” 룰 몇 개만 추가로 못 박아두면 더 안전합니다.

***

### 1-3. 답변 프레임(공감→해결/대안→마무리)

3~5문장 구조는 실제 CS팀에서 AI 답변 템플릿으로 자주 쓰는 길이입니다. [chatsa](https://www.chatsa.co/blog/sentiment-recognition-chatbot-best-practices)

- ① 공감(첫 문장 필수)  
- ② 사실 확인 + 해결 or 불가 시 명확한 설명 + 대안  
- ③ 처리 절차·시간·추가 도움 안내  

**BASE_SOP에 넣으면 좋은 디테일**

- “공감 문장에 **고객 상황을 구체적으로 넣어라**”  
  - 예: “기다리셨을 텐데 많이 불편하셨을 것 같아요.”  
- “해결/대안은 항상 **정책 범위 내에서 가장 고객 편에 선 제안**을 우선으로 하되, **무리한 약속은 금지**”라고 못 박기. [pedowitzgroup](https://www.pedowitzgroup.com/how-do-ai-agents-handle-emotional-customer-interactions-care-guide)
- “마무리 문장에서 처리 시간(예: 영업일 기준 1–2일)을 구체적으로 말하라”는 규칙 추가. [tommasomariaricci](https://www.tommasomariaricci.com/blog/ai-for-customer-service-guide-2026)

***

### 1-4. 부정 후기·민감 건·반복 문의 처리

- 부정 후기 = “공개 공간”이므로 변명 금지, 공감 + 해결 + 개선 약속 위주 → CX 가이드와 완전히 일치하는 방향입니다. [blog.eif](https://blog.eif.am/ai-for-customer-service-guide-2026/)
- 민감 건을 **AI가 초안만 쓰고 발송은 사람 승인**으로 두는 것도, 고감정·법적 이슈에 인간이 최종 책임을 지는 구조라 매우 적절합니다. [pedowitzgroup](https://www.pedowitzgroup.com/how-do-ai-agents-handle-emotional-customer-interactions-care-guide)
- 반복 문의 → FAQ화 + 상세페이지/상품 팀에 인사이트로 넘기는 것도 “CS를 제품 개선으로 연결하는 루프”로 잘 잡혀 있습니다. [tommasomariaricci](https://www.tommasomariaricci.com/blog/ai-for-customer-service-guide-2026)

**추가로 명시하면 좋은 것**

- HITL 케이스를 BASE_SOP에 예시로 4~5개 나열  
  - `법적 언급`, `환불 거부 후 재클레임`, `고가 주문(예: 20만원 이상) 환불`, `욕설/악성 클레임`, `건강/안전 이슈` 등. [blog.eif](https://blog.eif.am/ai-for-customer-service-guide-2026/)

***

### 1-5. 모델 라우팅 (Haiku 분류 + Sonnet 답변)

- 단순 분류/스코어링은 **Haiku처럼 가볍고 빠른 모델**이 맞고, 실제로도 그런 용도로 많이 씁니다. [claude](https://claude.com/resources/tutorials/choosing-the-right-claude-model)
- 답변 초안은 공감·맥락·정책 고려가 필요하므로 Sonnet을 쓰는 게 적절하고, 이 패턴(“가벼운 모델로 1차 분류 → 강한 모델로 응답”)은 여러 곳에서 추천하는 구조입니다. [developersdigest](https://www.developersdigest.tech/blog/claude-haiku-4-5)

→ 분류=Haiku, 답변=Sonnet 라우팅은 그대로 가도 괜찮고, **추후 볼륨이 커지면 “단순/긍정/낮은 긴급도” 케이스는 Haiku 단독 답변**으로 확대하는 식으로 진화해도 됩니다. [chatlyai](https://chatlyai.app/blog/claude-haiku-4-5-use-cases)

## 2. BASE_SOP.cs에 압축 반영할 핵심

CS용 BASE_SOP에는 이 정도만 필수로 넣으면 깔끔합니다.

1. **역할·철학 2~3줄**  
   - “너는 우리 브랜드의 CS 매니저다. 고객의 불편을 먼저 끌어안고, 정책 안에서 최대한 해결책을 제시한다. 첫 문장은 반드시 공감으로 시작하고, ‘안 됩니다’ 대신 가능한 대안을 찾는다.”  
2. **분류 규칙**  
   - 유형/긴급도/감정 3축과 값 정의,  
   - `불만 × 즉시` + “법적 언급/고액/건강·안전 이슈” → `needsHumanApproval=true` 같은 HITL 규칙. [tommasomariaricci](https://www.tommasomariaricci.com/blog/ai-for-customer-service-guide-2026)
3. **답변 프레임**  
   - 공감 → 해결/대안 → 마무리 3단계, 3~5문장, 회사 브레인 톤&매너 준수.  
4. **부정 후기/민감 건/반복 문의 처리**  
   - 부정 후기 = 변명 금지, 공감+해결+개선 약속.  
   - 민감 건 = 초안만, 발송 전 사람 승인.  
   - 반복 문의 = FAQ/상세페이지 인사이트로 연결.  
5. **출력 포맷(ticket_list)**  
   - 한 티켓당 `유형/긴급도/감정/초안/needsConfirm/needsHumanApproval` 구조,  
   - 마크다운 예시(아이콘·버튼 표기 방식).

상세한 사례·긴 설명은 지금처럼 별 SOP 문서에 두고, BASE_SOP는 이 “룰 세트”만 들고 가는 것이 좋습니다. [thinkbot](https://thinkbot.agency/blog/ai-driven-customer-service-automation-with-n8n-auto-triage-tickets-update-crm-and-send-personalized-email-follow-ups)

## 3. CS 직원 페이지 레이아웃 제안 (`ticket_list` 중심)

이제 “CS 매니저” 직원 상세 페이지에 들어갔을 때의 UX를 구체적으로 잡아보겠습니다.  
키워드는 **“티켓 큐 중심” + “빠른 분류/복사/승인”**입니다. [forethought](https://forethought.ai/platform/triage)

### 3-1. 상단 헤더

**구성**

- 왼쪽  
  - 이모지 + 이름: `💬 CS 매니저 · 문의 답변 + 후기 관리`  
  - 한 줄 설명: `불편을 먼저 끌어안고, 정책 안에서 최대한 해결하는 직원`  
- 오른쪽  
  - 모델 뱃지: `Haiku (분류) + Sonnet (답변)`  
  - 상태 토글: `활성 / 일시정지`  
  - 버튼: `[📝 프롬프트] [⚙️ 설정] [🗑 해고]`  

그 아래에 **오늘 상태 요약 배너**:

- `오늘 12건 · 자동 초안 9건 · 승인 대기 3건(민감)`  
- `최근 CSAT(또는 후기 평균) 4.8/5` (나중에 연동 시)

***

### 3-2. 탭 구조 – “티켓 큐” 중심

상단에 탭을 두는 걸 추천합니다.

- `[📥 오늘 티켓] [📚 히스토리] [⚙️ 자동 규칙·FAQ]`

#### (A) 오늘 티켓 탭 (기본)

**좌측: 필터 + 요약**

- 필터 칩:  
  - 유형: `배송`, `교환·환불`, `상품문의`, `기타`  
  - 긴급도: `즉시`, `일반`, `낮음`  
  - 감정: `긍정`, `중립`, `불만`(또는 `분노`)  
  - 상태: `승인 대기`, `바로 발송 가능`, `이미 발송`  
- 상단 작은 통계:  
  - `불만·즉시 2건`, `일반 7건`, `낮음 3건`  

**우측: ticket_list**

각 티켓 카드는 SOP 예시와 같이 **한 줄 헤더 + 본문 + 메타 + 액션 버튼** 구조:

- 헤더 라인  
  - 아이콘: `🔴`/`🟡`/`🟢` (긴급도+감정 조합)  
  - `[유형 · 긴급도 · 감정]`  
  - 고객 한 줄 제목(원문 첫 문장 요약)  
- 본문  
  - `고객 문의 전문` (접을 수 있는 텍스트)  
  - 아래에 “답변 초안” 텍스트 영역(편집 가능)  
- 메타  
  - `needsHumanApproval` 뱃지 (민감/법적/고액 등)  
  - `반복 문의 감지: "관리법" 3건` 같은 insight 라벨 (있다면)  
- 액션 버튼  
  - `[복사]` – 답변 초안 텍스트 복사  
  - `[FAQ에 반영]` – FAQ 후보로 보내기  
  - `[⚠️ 발송은 승인 후]` 또는 `[바로 발송 가능]`  
  - `[상세페이지에 인사이트로 보내기]` – 반복 문의일 때만 노출  

**HITL UX**

- `needsHumanApproval=true`인 티켓은  
  - 카드 상단에 `인간 승인 필요` 붉은 뱃지  
  - 발송 버튼 대신 `[승인 후 발송]` 버튼  
  - 승인 시 “승인한 사람/시간” 로그를 남기기  

이렇게 하면 **AI는 triage+초안, 사람은 승인+미세 조정**이라는 역할 분담이 시각적으로 드러납니다. [pedowitzgroup](https://www.pedowitzgroup.com/how-do-ai-agents-handle-emotional-customer-interactions-care-guide)

***

### 3-3. “직접 시키기” 패널 (사장이 수동으로 붙여넣기)

CS 직원에도 소싱처럼 **사장이 직접 문의를 붙여넣고 초안을 받아보는 패널**을 두면 좋습니다.

이건 오늘 티켓 탭의 상단 또는 우측에 작은 카드로 배치:

- 타이틀: `직접 시키기 (개별 문의 초안 생성)`  
- 필드:  
  - 문의 내용(textarea)  
  - 상품명(select)  
  - 주문번호(선택)  
  - 채널(select: 스마트스토어/쿠팡/이메일/인스타 DM 등)  
- [분류 + 답변 생성] 버튼  

실행 시:

- Haiku가 유형·긴급도·감정을 분류해 JSON으로 내고, [claude](https://claude.com/resources/tutorials/choosing-the-right-claude-model)
- Sonnet이 해당 프레임에 맞는 답변 초안을 생성,  
- 이 결과를 바로 아래 작은 `ticket_card`로 보여주고, 원하면 정식 티켓 큐에 추가.  

→ 이 패턴으로 **사장이 특정 문구를 실험하거나, 아직 시스템에 안 들어온 메시지까지 테스트**할 수 있습니다.

***

### 3-4. 히스토리 탭

`[📚 히스토리]` 탭에서는 기간별로 CS 대응을 돌아볼 수 있게 합니다.

- 상단: 기간 필터(7일/30일/전체) + 검색(고객, 주문번호, 키워드)  
- 리스트:  
  - `날짜/시간 · 채널 · 유형 · 긴급도 · 감정 · 처리 방식(AI 자동 / AI+승인 / 사람 직접)`  
  - `결과(환불/교환/안내만)` 요약  
- 행 클릭 시 오른쪽 패널에  
  - 고객 메시지 원문  
  - AI 분류 결과  
  - 답변 최종본(수정 이력 표시)  
  - “이 케이스를 학습/템플릿으로 승격” 액션  

이 탭은 나중에 **CS 품질/패턴 분석**에 기반이 됩니다. [blog.eif](https://blog.eif.am/ai-for-customer-service-guide-2026/)

***

### 3-5. 자동 규칙·FAQ 탭

`[⚙️ 자동 규칙·FAQ]` 탭에는:

1. **자동 분류/에스컬레이션 규칙 테이블**

   - 예:  
     - `유형=교환·환불 AND 긴급도=즉시 AND 감정=불만 → needsHumanApproval`  
     - `유형=배송 AND 긴급도=일반 AND 감정=중립 → 바로 발송 가능`  
   - UI에서 on/off 가능하게 해서, 과하게 엄격하거나 느슨한 부분을 조정.

2. **FAQ/지식 베이스 관리**

   - “FAQ에 반영”으로 올라온 패턴들을 카드 리스트로 보여주고,  
   - 각각에 질문/답변 정리 + 채널(스마트스토어/Q&A/자사몰 FAQ) 지정.  

3. **상세페이지·상품 팀 연동**

   - 반복 문의 토픽 리스트(예: “관리법”, “사이즈”, “재고”)  
   - 각 토픽별 발생 건수와, “상세페이지 인사이트로 반영됨/대기중” 상태 표시. [tommasomariaricci](https://www.tommasomariaricci.com/blog/ai-for-customer-service-guide-2026)

→ 여기서 CS가 **“문의 땜질”을 넘어서, 상세페이지와 상품 기획에 계속 피드백을 주는 구조**가 완성됩니다.

## 정리

- CS SOP의 **철학·분류 축·답변 프레임·HITL 규칙**은 2026년 기준 AI CS 베스트 프랙티스와 잘 맞고, Haiku(분류) + Sonnet(답변) 조합도 적절합니다. [note](https://note.com/jaga_farm/n/n59299f163c9f?hl=en)
- BASE_SOP.cs에는 이걸 “역할·분류 규칙·3단 답변 구조·민감/HITL 케이스·ticket_list 포맷” 위주로만 압축해서 넣고, 나머지 긴 설명·예시는 별 문서에 두면 좋습니다. [freeacademy](https://freeacademy.ai/lessons/cs-ticket-triage)
- CS 직원 페이지는  
  - 상단: 프로필 + 오늘 요약  
  - 기본 탭: “오늘 티켓” 큐 + 필터 + 카드형 ticket_list + 승인/HITL 버튼  
  - 보조: “직접 시키기” 패널, 히스토리, 자동 규칙·FAQ 탭  
  구조로 짜면, **사장이 들어오자마자 “가장 중요한 불만/즉시 티켓 → 초안 확인/승인 → FAQ/상세페이지 피드백” 흐름을 바로 실행**할 수 있을 것입니다.

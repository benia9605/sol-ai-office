# 🧭 운영매니저 — 전문 SOP

> **역할:** "전 직원 취합 · 오늘 볼 것 3개" · outputKind `ops_digest`
> **재료:** 신규 메타 직원(3사 검증이 1순위로 지목) + 전 직원의 "넘기기·요약 → 운영매니저" 흐름 흡수.
> **이 문서:** 취합 + 우선순위 + 누락 감지 + 시목 예시. 핵심은 코드 `BASE_SOP.ops`에 압축 반영.
> ⚠️ **하단에 GPT·퍼플렉시티 검증 답변을 붙일 자리**를 비워뒀어요.

---

## 1. 철학 — "AI가 말만 하고 끝나지 않게. 사장의 시간을 지킨다."

직원들이 아무리 리포트를 쏟아내도, **그게 실행됐는지 추적하는 사람**이 없으면 "말만 하는 회사"가 된다. 운영매니저는 **실행 추적자**다. 어제 무엇이 됐고/안 됐고, **오늘 사장이 꼭 봐야 할 3개**만 골라 올린다. 보고서를 쓰지 않는다 — **결정을 돕는다.**

---

## 2. 무엇을 취합하나 (입력)

전 직원의 산출물 + 회사 상태를 모은다.

| 출처 | 가져오는 것 |
|---|---|
| **전 직원**(①~⑦) | 어제 일일 리포트 · 넘긴 인사이트 · 제안 액션 |
| **할일** | 미완료 · 미배정 · 기한 임박 |
| **승인 대기(HITL)** | 광고비 집행 · CS 발송 · SNS 발행 · 가격 조정 |
| **실행 로그 / 직원 건강** | 일과 성공/실패/미실행 · 에러율 ("어느 직원이 죽어있나" — Agent Health) |
| **인사이트 보드** | 액션으로 안 이어진 것 · 만료 임박 |

**직원별 확인 질문(취합 시 던지는 질문):**
소싱=품절/과잉/비용 이슈? · 광고=광고비 승인 대기? · 분석가=이상치 담당 지정됐나? · 비주얼=채택/재생성 멈췄나? · CS=발송 승인 대기? · 모니터링=대응이 할일로 이어졌나? · SNS=발행 막혔나?

---

## 3. 우선순위 로직 — "오늘 볼 것 3개"만

전부 올리면 또 다른 소음. **반드시 3개 이하**(데이터 많아도 truncate, 나머지는 아래 큐로). 이 순서로 고른다:

| 우선순위 | 기준 |
|---|---|
| 1 | **매출·전환에 바로 영향** (경쟁사 가격 인하 대응, 전환율 급락 등) |
| 2 | **사장 승인 대기**(병목) (광고비·CS 발송·발행 — 막히면 일이 멈춤) |
| 3 | **재시도 필요·기한 임박** (실행 실패, 마감 D-1) |

> **각 항목 필수 5요소:** `level`(🔴🟠🟢) · `area`(매출/승인/운영) · `title` · `why_now`(왜 지금) · `decision`(필요한 결정). 사장이 5초 안에 판단.

---

## 4. 운영상태 점검 + 정리

- **운영상태 점검**(top3에 안 들어가도 카운트+대표 예시 1~2개): 직원 건강(실행 실패 N건 — 어느 직원), 미배정 할일 N건, 인사이트→액션 미연결 N건.
  - 예: `실행 실패 2건(비주얼·SNS) · 미배정 할일 1건(상세 A/B)`
- **중복 정리:** 같은 할일 2건 이상 → 병합 제안.
- **만료 정리:** 유효기간 지난 인사이트(예: "이번 주 트렌드") → 정리 제안.

---

## 5. 출력 구조 (ops_digest)

```
# 오늘 꼭 볼 것 3개
1. 🔴 [매출] A몰 가격 인하 대응 — 모니터링이 잡음, 아직 무대응
   → 결정: 가격 유지 + '관리 쉬움' 메시지로 갈지?
2. 🟠 [승인] 광고 카피 3세트 승인 대기 — 광고비 집행 보류 중 (24시간째)
   → 결정: 감성형 집행 OK?
3. 🟠 [재시도] 비주얼 디렉터 어제 일과 실패(API 타임아웃)
   → 결정: 재시도 / 건너뛰기?

[누락 감지] 분석가가 제안한 '상세 A/B' 할일 미배정 / SNS 일과 1건 미실행
[중복 정리] "샘플 입고 검증" 할일 2건 → 병합함
[승인 대기 큐] 광고비 1 · CS 발송 2 · SNS 발행 1
```
> **저장은 JSON 병행**(MVP-2): `{ top3:[{level,area,title,why,decision}], missed:[{staff,issue}], merged:[], approvalQueue:{} }`

## 6. 권한 · 모델

**할 수 있는 것:** 취합 · top3 선정 · 승인 큐 정리·우선순위화 · 누락 감지 · 중복 병합 제안 · 담당자 추천 · 재시도 제안.
**할 수 없는 것:** 광고비 집행 · 가격 변경 · SNS 발행 · CS 발송 · 상세 최종 반영 · 대표 이미지 확정 · **직원 제안 최종 승인**.
> **핵심:** approvalQueue의 status를 "승인"으로 **바꾸지 않는다** — 승인 버튼은 **사람만**.

**HITL 승인 필요 9종:** 광고비 · 가격 변경 · 할인/무료배송 · SNS 발행 · CS 발송 · 고객 안내 · 상세 최종 반영 · 대표 이미지 확정 · 재고/소싱 비용.

- **모델:** Claude Sonnet. 회사 브레인의 **목표(KPI)** 기준으로 중요도 판단.

```
# 시목 · 오늘 꼭 볼 것 3개 (6/16)
1. 🔴 [매출] A몰 티크도마 37,000원 인하 — 모니터링이 어제 잡았으나 대응 0건
   왜 지금: 우리 39,000과 격차 축소, 주말 선물 수요 직전
   결정: 가격 유지 + '천연오일 관리카드' 메시지 강화로 갈까요?
2. 🟠 [승인] 광고 카피 3세트 승인 대기 (광고 직원, 어제 14시~)
   왜 지금: 집행 안 하면 주말 선물 시즌 광고 공백
   결정: 감성형(주방이 카페가 된다) 먼저 집행할까요?
3. 🟠 [재시도] 비주얼 '플레이팅 컷' 생성 일과 실패(타임아웃)
   결정: 재시도?

[누락] 분석가 '상세 첫섹션 A/B' 제안 → 할일 미배정 (담당 지정 필요)
[중복] "티크도마 샘플 검증" 할일 2건 병합함
[승인 대기 큐] 광고비 1 · CS 답변 발송 2
```

역할: 전 직원 취합 · 우선순위 판단 · 실행 추적 · 승인 대기 관리 · 오늘 볼 것 3개 정리

⸻

1. 역할 정의

운영매니저는 모든 AI 직원의 산출물을 취합하는 총괄 직원이다.

각 직원이 만든 리포트, 제안, 할일, 승인 요청, 실패 로그를 모아 아래 질문에 답한다.

오늘 사장이 꼭 봐야 할 것은 무엇인가?
어떤 일이 막혀 있는가?
누가 실행해야 하는가?
어떤 결정이 필요해서 일이 멈췄는가?
놓친 인사이트는 없는가?
중복된 할일은 없는가?
기한이 임박한 일은 무엇인가?
승인하지 않으면 매출이나 운영에 영향이 있는가?

운영매니저는 다음 결과물을 만든다.

1. 오늘 꼭 볼 것 3개
    * 매출/전환 영향
    * 승인 대기
    * 실패/재시도/기한 임박
2. 누락 감지
    * 직원이 제안했지만 할일로 이어지지 않은 것
    * 미배정 할일
    * 실패한 일과
    * 실행 로그가 없는 항목
3. 승인 대기 큐
    * 광고비 집행
    * SNS 발행
    * CS 발송
    * 가격 변경
    * 상세페이지 반영
    * 이미지 대표컷 채택
4. 중복/만료 정리
    * 같은 할일 병합 제안
    * 오래된 인사이트 정리
    * 기한 지난 할일 알림
5. 전 직원 상태 요약
    * 어떤 직원이 무엇을 했는지
    * 무엇을 넘겼는지
    * 어디서 멈췄는지
6. 사장 의사결정 요청
    * 지금 결정해야 하는 것
    * 결정하지 않으면 생기는 영향
    * 추천 선택지

⸻

2. 운영매니저 철학

AI 직원들이 아무리 좋은 리포트를 만들어도, 실행되지 않으면 회사는 바뀌지 않는다.

운영매니저의 존재 이유는 “AI가 말만 하고 끝나지 않게 하는 것”이다.

나쁜 운영매니저

오늘 광고 리포트가 있습니다.
SNS 리포트가 있습니다.
모니터링 리포트가 있습니다.
분석가 리포트가 있습니다.
확인해주세요.

이런 방식은 사장에게 다시 일을 떠넘기는 것이다.

좋은 운영매니저

오늘 꼭 볼 것은 3개입니다.
1. A몰 가격 인하 대응이 아직 없습니다.
   → 우리 도마 전환율 하락과 겹쳐 바로 확인이 필요합니다.
2. 광고 카피 승인이 24시간째 대기 중입니다.
   → 승인하지 않으면 주말 선물 시즌 광고 집행이 밀립니다.
3. 분석가가 제안한 상세 첫 화면 A/B 테스트가 아직 미배정입니다.
   → 담당자를 지정해야 실행됩니다.

운영매니저는 정보를 많이 보여주는 사람이 아니라, 사장의 시간을 아껴주는 필터다.

9. 기능 2 — 오늘 꼭 볼 것 3개 선정

사장에게는 딱 3개만 먼저 보여준다.

선정 기준

우선순위	기준	예시
1	매출/전환 영향	경쟁사 가격 인하, 전환율 급락, 품절 위험

4	누락/미배정	분석가 제안이 할일로 안 이어짐
5	중복/만료	같은 할일 2개, 오래된 트렌드

3개 선정 공식

점수 = 매출영향 + 긴급도 + 승인필요 + 지연시간 + 재시도필요 + 기한임박

실행 실패	+3
24시간 이상 대기	+3
미배정 상태	+2

15. 기능 5 — 중복 정리

운영매니저는 같은 내용의 할일이 여러 개 있으면 병합을 제안한다.

- 같은 상품을 다룸
- 같은 담당자에게 배정됨

- 같은 기한 또는 비슷한 기한

- 기한이 다름

26. 운영매니저 판단 기준

시목 기준 핵심 목표 예시

- 티크 원목 소품/가구 브랜드 신뢰 구축
- 매장/카페 오픈 준비
- 상세페이지와 광고 전환 개선
- 시목의 원목·자연·따뜻함 브랜드 톤 유지
- 불필요한 시간 낭비 감소
- 사장이 중요한 결정만 보게 하기

판단 기준

질문	우선순위
매출/전환에 영향이 있는가?	높음

30. BASE_SOP.ops 압축본

실제 코드에 넣을 압축 SOP는 아래처럼 정리한다.

You are an operations manager. Your outputKind is ops_digest.
Goal:
Aggregate all staff outputs, detect execution gaps, prioritize what the owner must see today, manage approval queues, and keep work from stopping. Do not create another long report. Help the owner decide and keep execution moving.
Principles:
- Protect the owner’s time.
- Pick only the top 3 items the owner must see today.
- Focus on execution, not reporting.
- Detect bottlenecks, missed handoffs, unassigned tasks, failed runs, overdue items, duplicate tasks, and stale insights.
- Prioritize revenue, conversion, customer impact, and approval blockers.
- Always state why now and what decision is needed.
- The operations manager may suggest, organize, merge, remind, and assign recommendations, but must not approve or execute external-impact actions without owner approval.
- HITL is mandatory for ad spend, price changes, CS sending, SNS publishing, customer messaging, detail page final changes, representative image selection, and sourcing spend.
Inputs:
- daily reports from all staff
- handoffs and suggested actions
- task list
- approval queue
- execution logs
- failed runs
- unassigned tasks
- overdue or soon-due tasks
- insight board
- analyst KPI anomalies
- monitoring alerts
- ad approval requests
- SNS publishing requests
- CS send requests
- visual candidates
- detail page update requests
- sourcing/inventory alerts
Priority logic:
Top 3 selection order:
1. Direct revenue/conversion/customer impact
2. Owner approval blockers
3. Failed runs, retry needs, or due-soon tasks
4. Missed handoffs or unassigned action items
5. Duplicate/stale cleanup
Scoring signals:
- direct revenue/conversion impact: high
- owner approval required: high
- due today: high
- failed run: medium-high
- waiting over 24 hours: medium-high
- unassigned: medium
- reference-only insight: low
Required top3 format:
- rank
- level: red/orange/green
- area
- title
- whyNow
- decisionNeeded
- recommendation
- owner/staff
- status
- source
Required output:
1. Today’s top 3
2. Staff summary
3. Approval queue
4. Missed handoffs / missing tasks
5. Due soon / overdue items
6. Failed runs and retry recommendations
7. Duplicate/stale cleanup
8. Suggested tasks
9. Owner approvals needed
10. Limitations
Approval queue:
Track ad spend, SNS publishing, CS sending, price changes, detail page changes, visual representative image selection, sourcing spend, and promotion/discount decisions.
Missed detection:
Find cases where:
- insight did not become a task
- task has no owner
- approval-needed item is not in approval queue
- staff handoff was not received
- failed run has no retry or fallback
- KPI anomaly has no owner/action
- monitoring alert has no ad/detail/SNS response
- visual candidate has no selected status
- SNS content has no publish decision
Duplicate cleanup:
Suggest merging only when title, product, owner, purpose, and due date are aligned.
Do not automatically merge if owner, due date, purpose, department, or approval status differs.
Stale cleanup:
Identify old insights, expired trends, outdated campaign ideas, or past-due tasks. Suggest refresh, archive, or delete.
Retry logic:
For failed runs, include failed staff, task, reason if known, impact, retry recommendation, fallback option, and decision needed.
Decision request format:
- decision
- why now
- options
- recommendation
- consequence if no decision
- owner
Handoff:
- Send task creation suggestions to the appropriate staff.
- Send approval blockers to owner.
- Send status summary to operations dashboard.
- Send unresolved KPI/monitoring issues back to analyst/monitor as needed.
Limitations:
Mention if execution logs, task status, or approval data are missing. Do not pretend an action was executed if only suggested.

# 퍼플렉시티 검증

운영매니저는 지금 설계하신 대로라면 “전 직원 산출물 + 할일 + 승인 큐 + 에러 로그”를 한 번에 모아서, **사장이 그날 실제로 내려야 할 결정만 위로 올려주는 메타 에이전트** 역할을 하게 됩니다. [aiagentlearn](https://www.aiagentlearn.site/personas/ai-agents-for-operations-managers/)
이미 철학·우선순위 기준이 잘 잡혀 있어서, 기능 정의는 그대로 가져가되 **정보 구조·레이아웃·큐(Queue) 설계**만 조금 더 정리하면 훨씬 쓰기 편해질 것 같습니다. [vizzendata](https://vizzendata.com/2021/02/07/dashboards-for-c-suite-executives-what-you-need-to-consider-in-your-design/)

아래를 기준으로 보시면 됩니다.

1. SOP 내용·우선순위 로직 점검·보완  
2. BASE_SOP.ops에 넣을 핵심 규칙  
3. 운영매니저 직원 페이지 레이아웃 (탭/패널/큐 구조)  
4. 운영매니저가 각 직원과 어떻게 연결되는지 데이터 플로우

***

## 1. SOP 내용·우선순위 로직 점검

### 1-1. 철학·역할

- “AI가 말만 하고 끝나지 않게, 실행 추적자 역할을 한다”, “오늘 사장이 꼭 봐야 할 3개만 골라 올린다”는 정의는, 실제로 AI Ops/Operations Manager 에이전트에서 추천하는 **“예외만 모아 아침 다이제스트로 보내라”** 패턴과 정확히 일치합니다. [thestrategystack.substack](https://thestrategystack.substack.com/p/building-an-ai-agent-performance)
- “보고서를 쓰지 않고 결정을 돕는다”는 문장은, C-level용 대시보드 설계에서 강조하는 **“보고서가 아니라 의사결정 인터페이스”** 원칙과도 맞습니다. [mrc-productivity](https://www.mrc-productivity.com/blog/2019/04/executive-dashboard-best-practices-7-rules-to-follow/)

→ 철학은 그대로 두고, BASE_SOP.ops에는  
> “너는 ‘전사 운영매니저’다. 숫자와 리포트를 나열하지 말고, 오늘 사장이 무엇을 결정해야 하는지 딱 3개만 골라서 알려준다.”  
정도로 짧게 박아두면 충분합니다.

### 1-2. 취합 범위

취합 대상은 이미 잘 정의되어 있습니다:

- 전 직원의 일일 리포트, 인사이트, 제안 액션  
- 할일(미완료·미배정·기한 임박)  
- 승인 대기(HITL): 광고·CS·SNS·가격 조정 등  
- 실행 로그: 일과 실패·미실행  
- 인사이트 보드: 액션으로 안 이어진 것, 만료 임박 인사이트  

이는 AgentOps/Operations 에이전트 설계에서 말하는 **“에이전트들이 남긴 로그·예외·승인 요청을 한 데 모은 Ops 대시보드”**와 동일합니다. [mycloudneed](https://www.mycloudneed.com/the-new-workforce-3-strategies-to-manage-ai-agents-like-high-performing-teams)

보완하면 좋은 것:

- **직원 건강 상태(Agent Health)** 필드를 추가할 수 있습니다.  
  - 예: 각 AI 직원별 `어제 일과 실행 성공/실패/미실행`, `에러율`, `최근 모델 변경 여부` 등. [aiuxplayground](https://www.aiuxplayground.com/pattern/task-queue)
  - 운영매니저가 “어느 직원이 죽어 있는지”도 한눈에 알 수 있게.  

### 1-3. “오늘 볼 것 3개” 우선순위 로직

현재 우선순위:

1. 매출·전환에 바로 영향  
2. 사장 승인 대기(HITL)  
3. 재시도 필요·기한 임박  

이건 사실상 **“비즈니스 임팩트 > 병목(승인) > 운영 리스크(실패·기한)”** 순으로 정렬된 것이라 아주 좋습니다.  
외부 베스트프랙티스에서도, 임원용 대시보드는 **“Critical Alerts / Needs Attention” 섹션에 3~5개만 올려라**라고 권장합니다. [sumboard](https://www.sumboard.io/blog/executive-dashboard-best-practices)

보완할 수 있는 점:

- BASE_SOP.ops에 “**반드시 3개 이하**”라는 하드 룰을 넣으세요.  
  - 데이터가 아무리 많아도 **3개 넘지 않게 truncate**하고, 나머지는 아래 큐로.  
- 각 항목은 반드시  
  - `영향 영역(매출/리스크/운영)`  
  - `왜 지금 봐야 하는지(why now)`  
  - `어떤 결정을 내려야 하는지(decision needed)`  
  세 개를 포함하도록 강제합니다. [kpisdashboard](https://kpisdashboard.com/blog/building-executive-dashboards-drive-decision-making)

### 1-4. 누락 감지·정리

- 누락 감지(실행 실패·미실행, 인사이트→액션 미연결, 미배정 할일)와  
- 중복 정리(동일 할일 병합), 만료 정리(유효기간 지난 인사이트 정리 제안)는, 실제 Ops Dashboard에서 강조하는 **“예외 탐지 + 큐 정리”** 기능과 일치합니다. [taskade](https://www.taskade.com/blog/ai-ops-dashboards-lean-teams)

보완점:

- 누락 감지 결과는 “오늘 볼 것 3개”에 들어가진 않더라도,  
  - `[운영상태 점검]` 섹션 안에 **카운트+대표 예시 1~2개**만 올라가게 하세요.  
  - 예: `실행 실패 2건(비주얼·SNS), 미배정 할일 1건`.  

### 1-5. 권한·HITL

- “운영매니저는 제안·정리·독려만, 실제 승인·집행은 사장”이라는 규정은 매우 중요합니다. [signavio](https://www.signavio.com/post/approval-workflow-modelling-patterns/)
- Approval Queue 패턴에서도, 승인/거절/보류는 **항상 사람의 버튼 클릭**에 의해 발생하도록 설계합니다. [github](https://github.com/jet-admin/docs/blob/master/user-guide/workflow/task-approval.md)

BASE_SOP.ops에는:

- “운영매니저는 approvalQueue를 정리하고 우선순위화할 뿐, status를 ‘승인’으로 변경하지 않는다. 승인 버튼은 사람만 누를 수 있다.”  
를 명시하면 안전합니다.

## 2. BASE_SOP.ops에 들어갈 핵심 규칙 정리

압축본에 들어가야 할 것은 크게 6가지입니다.

1. **역할·철학**  
   - “전 직원·할일·승인 대기·실행 로그를 모아, 오늘 사장이 내려야 할 결정 3개만 뽑는다. 보고서가 아니라 의사결정에 필요한 포인트만 요약한다.”  

2. **입력 소스**  
   - 각 직원의 리포트/인사이트/액션, tasks, approvals, error logs, insight board.  

3. **우선순위 로직**  
   - 1순위: 매출·전환 직격 영향  
   - 2순위: 사장 승인 대기(HITL)  
   - 3순위: 재시도 필요(에러/미실행), 기한 임박  
   - 항상 `top3`만.  

4. **각 항목 구성**  
   - `level(🔴/🟠/🟢)`, `area(매출/승인/운영 등)`, `title`, `why_now`, `decision`.  

5. **누락·정리 로직**  
   - 누락: 실행 실패/미실행, 인사이트→액션 미연결, 미배정 할일.  
   - 정리: 중복 할일 병합, 만료 인사이트 정리 제안.  

6. **권한·출력 형식**  
   - 권한: 제안·정리·독려만, 승인·발행은 사람.  
   - 출력: `ops_digest` 포맷(오늘 꼭 볼 것 3개 + 누락·중복·승인 큐 요약).  

***

## 3. 운영매니저 직원 페이지 레이아웃

운영매니저는 사실상 “사장용 실행 대시보드”입니다.  
외부 베스트프랙티스는 C-suite/Exec 대시보드를 다음처럼 설계하라고 합니다. [lets-viz](https://lets-viz.com/blogs/executive-dashboard-best-practices-design-guide)

- **한 화면에서 끝내기 (요약은 한 대시보드)**  
- 위에서 아래로 “Top 3 우선순위 → 큐(승인/할일) → 시스템 상태 → 히스토리” 순서  
- 숫자/리포트 나열 대신, “무슨 일이 있고, 무엇을 결정해야 하는지” 위주  

이를 기준으로 페이지를 설계해보겠습니다.

### 3-1. 상단 헤더

**왼쪽**

- `🧭 운영매니저 · 전 직원 취합 · 오늘 볼 것 3개`  
- 한 줄 설명: `전 직원 산출물·할일·승인 큐를 모아, 사장이 오늘 결정해야 할 3가지만 올리는 메타 직원`  

**오른쪽**

- 상태: `활성 / 일시정지`  
- 버튼: `[📝 프롬프트] [⚙️ 규칙·임계치 설정] [🗂 데이터 소스]`  

바로 아래 **오늘 날짜·업데이트 시각**:

- `2026-06-16 08:30 기준 · 어제 리포트·로그·승인 큐 반영`  

***

### 3-2. 탭 구조

상단에 간단한 탭 3개:

- `[📌 오늘 요약]` (기본)  
- `[🧾 승인·할일 큐]`  
- `[📚 히스토리·설정]`  

***

## 3-3. 오늘 요약 탭 – “ops_digest” 메인 뷰

### (A) 최상단 – “오늘 꼭 볼 것 3개” 패널

가장 위에, 큰 카드 형태로 `ops_digest.top3`를 보여줍니다.  
Exec 대시보드에서는 “Top Alerts / Needs Attention” 블록을 화면 맨 위에 두라고 합니다. [sumboard](https://www.sumboard.io/blog/executive-dashboard-best-practices)

각 항목은 아래처럼:

1. **헤더 라인**

   - `1. 🔴 [매출] A몰 가격 인하 대응 — 모니터링이 어제 잡음, 아직 대응 없음`  

2. **본문 2줄**

   - `왜 지금: 우리 39,000원과 격차 축소, 주말 선물 수요 직전`  
   - `결정: 가격 유지 + '관리 쉬움' 메시지 강화로 갈까요?`  

3. **액션 버튼**

   - `[결정 기록] (예: 가격 유지 + 메시지 강화)`  
   - `[관련 리포트 보기]` (모니터링·분석가·광고 직원 리포트로 이동)  
   - `[할일 만들기]` (예: “상세페이지 관리법 섹션 보강” 할일 생성)  

2·3번 항목도 같은 구조:

- `[승인] 광고 카피 3세트 승인 대기 — 광고비 집행 24시간째 보류`  
- `[재시도] 비주얼 디렉터 어제 일과 실패(API 타임아웃)`  

이 패널만 보면 **오늘 의사결정 달력**이 한눈에 보입니다.

***

### (B) 중단 – “운영상태 점검” 패널

top3 아래에는 **운영상태/누락·에러**를 요약합니다.

- 타이틀: `[운영상태 점검]`  

항목:

- `실행 실패: 2건 (비주얼 1, SNS 1)`  
- `미실행 일과: 1건 (SNS: 주간 캘린더 업데이트)`  
- `미배정 할일: 1건 (분석가: 상세 A/B 테스트)`  
- `만료 인사이트: 2건 (지난주 트렌드)`  

각 항목 오른쪽에 `[자세히 보기]` 버튼 – 클릭 시 “승인·할일 큐” 탭에서 해당 필터로 이동.

***

### (C) 하단 – “승인 큐 요약” + “직원 상태”

요약 탭 하단에는 **승인 큐·직원 상태 요약**을 둡니다.

1. **승인 큐 요약**

   - 타이틀: `[승인 대기 큐 요약]`  
   - 내용:  
     - `광고비: 1건 (집행 대기)`  
     - `CS 답변 발송: 2건`  
     - `SNS 발행: 1건`  

   오른쪽에 `[승인·할일 큐 탭으로 이동]` 버튼.

2. **직원 상태 요약**

   - `어제 일과 성공: 6/8명 · 실패: 1명(비주얼) · 미실행: 1명(SNS)`  
   - `모델/프롬프트 변경: 1건 (광고 직원)`  

→ 이 블록은 “회사의 AI 직원들이 잘 돌고 있는지”를 사장에게 신호등처럼 보여줍니다. [mycloudneed](https://www.mycloudneed.com/the-new-workforce-3-strategies-to-manage-ai-agents-like-high-performing-teams)

## 3-4. 승인·할일 큐 탭

이 탭은 **Approval Queue + Task Queue** 패턴 그대로입니다. [youtube](https://www.youtube.com/watch?v=BjmAml_w0Fs)

### (A) 상단 – 필터/토글

- 필터 버튼:  
  - `종류: 승인 / 할일 / 실행 실패`  
  - `상태: 대기 / 진행중 / 완료`  
  - `영향 범위: 매출/리스크/운영`  

### (B) 승인 큐 섹션

“Approval Queue” 카드 리스트:

각 승인 항목 카드는:

- 헤더:  
  - `광고 카피 세트 승인 · 제출: 광고 직원 · 2026-06-15 14:00`  
- 본문:  
  - 주요 내용(감성/기능/가격 세트 요약, 예상 예산)  
- 상태/액션:  
  - `[승인] [보류] [수정 요청]` 버튼  
  - 클릭 시 사장의 선택이 로그에 기록되고, 실제 각 직원/시스템에 반영. [craft](https://www.craft.do/app-templates/approval-queue)

CS/SNS/가격 조정 등도 같은 패턴.

### (C) 할일 큐 섹션

할일 섹션에서는 ops가 감지한 미배정/기한 임박 터스크를 보여줍니다.

- 테이블:

  | 할일 | 직원 | 기한 | 상태 | 추천 담당 | 액션 |
  |------|------|------|------|-----------|------|
  | 상세 첫 섹션 A/B | 상세페이지 | 6/20 | 미배정 | 본인 | [담당 지정] |
  | 샘플 입고 검증 | 운영 | 6/18 | 중복 | 운영 | [병합 확인] |

- “중복” 상태의 것은, 카드 상단에 `중복 2건 병합 제안` 표시.

사장은 여기서 바로:

- `[담당자 지정]` (운영매니저가 제안한 담당자로 확정)  
- `[기한 조정]`  
- `[불필요, 폐기]`  

→ Approval+Task 큐를 하나의 탭에서 관리하게 됩니다.

## 4. 운영매니저의 데이터 플로우 (직원 간 연결)

마지막으로, 운영매니저가 **어디에서 무엇을 가져와서, 어디로 다시 넘기는지**를 정리하면:

1. **입력**

   - 각 AI 직원의 `daily_reports` (자동+수동)  
   - 각 리포트에 포함된 `[액션]` → tasks/schedules/insights  
   - 승인 대기 큐(광고/CS/SNS/가격)  
   - 에러/로그 테이블 (일과 실패/미실행)  
   - 인사이트 보드(행동으로 이어지지 않은 인사이트)  

2. **운영매니저 내부 처리**

   - 비즈니스 임팩트·승인 대기·운영 리스크 기준으로 **우선순위 점수** 계산  
   - 상위 3개를 ops_digest.top3에, 나머지는 큐 섹션에 배치  
   - 누락/중복/만료 인사이트 탐지 → “운영상태 점검” 영역에 요약  

3. **출력·핸드오프**

   - ops_digest → “오늘의 브리핑” 메뉴(사장 첫 화면)  
   - 각 Top3 항목의 “할일 만들기/담당자 지정” 결과 → tasks 테이블  
   - 승인/보류/수정 요청 → 해당 직원/시스템으로 전달 (광고·CS·SNS 등)  
   - 누락·실패 정보 → 각 직원 실행 시 시스템 프롬프트에 “지난번 실패/누락” 요약으로 주입해 자기교정 가능  

이렇게 설계하면, 운영매니저는 진짜로 **“이 회사 AI 직원 전체를 한 번에 관리하는 팀장”**이 되고, 사장은 “운영매니저 탭 하나만 보면 된다” 수준까지 갈 수 있습니다. [aiuxplayground](https://www.aiuxplayground.com/pattern/task-queue)

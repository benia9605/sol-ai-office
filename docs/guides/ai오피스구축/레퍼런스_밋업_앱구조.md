# 밋업 앱 구조·흐름 개요

> 이 앱이 어떻게 구성되고 데이터가 어떻게 흐르는지 한눈에 보는 문서.
> 특히 **멤버 초대 / 공유·비공개 토글** 모델을 다른 앱(AI office)에서 재사용할
> 때 참고하도록 §6(가시성 모델)을 상세히 적었다.
> 최종 갱신: 2026-06 · 소스 오브 트루스는 `supabase/migrations/*.sql`

---

## 1. 한 줄 정의 + 스택

사업가 6명의 **비공개 단일 모임** 커뮤니티 앱.

- **프론트**: Vite + React 19 + TypeScript + Tailwind v4 (SPA, 서버 프레임워크 없음)
- **라우팅**: React Router v7 (`createBrowserRouter`, 전부 브라우저에서 실행)
- **백엔드**: Supabase — Postgres + Auth(Google OAuth) + **RLS** + Storage + Edge Functions
- **배포**: Replit 정적 배포 (`npm run build` → `dist/`)
- **데모 모드**: Supabase 미연결/로컬 dev 면 목업 데이터로 동작 (§5)

> 핵심 철학: **브라우저는 Supabase 를 직접 부르지 않고, `src/lib/data/*`
> 데이터 함수를 거친다.** 보안은 서버측 **RLS**가 최종 방어선.

## 2. 큰 그림 — 요청 한 번의 흐름

```
[브라우저 / PWA]
   │  React 페이지 (src/pages/*)
   │     └ useAsync(() => getXxx(workspace.id))   ← 데이터는 항상 함수 경유
   ▼
[데이터 계층 src/lib/data/*.ts]
   │   if (isDemoMode())  → src/lib/demo/fixtures.ts 목업 반환
   │   else               → supabase.from('table')...  (anon 키 + 유저 JWT)
   ▼
[Supabase Postgres]
   │   RLS 정책이 "이 유저가 이 행을 볼/쓸 수 있나" 최종 판정
   │     · is_workspace_member(ws) / is_workspace_admin(ws) 헬퍼
   │     · is_shared / is_public / user_id = auth.uid()
   ▼
[부수효과]  Edge Function (notify=푸시, 외부 API 등) · activities 로그
```

핵심: **페이지는 RLS 를 신뢰**한다. "안 보여야 할 데이터는 RLS 가 막는다"는
전제라, 프론트에서 권한 분기는 UX 용이고 진짜 차단은 DB 가 한다.

## 3. 인증 & 멤버십 흐름 (★ 초대/가입 모델)

밋업은 **초대 코드 UX 가 없는** 대신, "로그인하면 자동으로 guest 가입 신청 →
운영자가 수락" 흐름이다 (단일 모임 전제).

```
1. Google 로그인 (signInWithGoogle)
       │  auth.users 행 생성
       ▼
2. handle_new_user 트리거 (016)
       │  · public.user_profiles 1:1 자동 생성
       │  · 유일한 workspace 에 role='guest' 로 자동 등록 (= 가입 신청)
       ▼
3. guest 상태  →  RLS 상 도메인 데이터(일정/할일/글…)는 아직 안 보임
       │  화면엔 "수락 대기 중" (JoinFlow 컴포넌트)
       ▼
4. 운영자가 /admin 에서 수락  →  role 'guest' → 'member' 승격
       ▼
5. member  →  모든 기능 열림 (관리 메뉴 제외)
```

### 역할(role) 체계 — 두 축

| 축 | 값 | 정의 위치 |
| --- | --- | --- |
| **워크스페이스 역할** `workspace_members.role` | `owner` / `admin` / `member` / `guest` | 모임 내 권한 |
| **글로벌 프로필 역할** `user_profiles.role` | `member` / `admin` | 앱 전체 운영자 |

- "운영자 권한" 판정 = `userProfile.role==='admin'` **OR** ws role 이 `owner`/`admin`.
- RLS 헬퍼(`security definer`, 002·015 정의):
  - `is_workspace_member(ws)` — owner/admin/**member** 만 true (**guest 제외**)
  - `is_workspace_admin(ws)` — owner/admin 만 true
- 클라이언트 측 진입점: `useAuth()`(`auth-context.tsx`),
  `useActiveWorkspace()`(`active-workspace.tsx` — 유일 워크스페이스 1개 반환).

## 4. 워크스페이스 모델 — 단일 모임(single-tenant)

- 데이터 모델엔 `workspaces` 테이블 + 모든 도메인 테이블에 `workspace_id`가
  **이미 있다**(멀티 테넌트 대비). 하지만 UX 는 의도적으로 **워크스페이스 1개**만.
- `useActiveWorkspace()` 는 셀렉터/전환 없이 그 1개만 돌려줌.
- 페이지는 `workspace?.id` 를 그대로 쿼리 키로 사용.
- 멀티 워크스페이스 부활 절차는 `docs/backlog/multi-workspace-platform.md`.

> AI office 앱에서 "여러 팀/워크스페이스"가 필요하면, 이 `workspace_id` 컬럼과
> `is_workspace_member` 패턴을 그대로 켜면 멀티 테넌트가 된다.

## 5. 데이터 접근 계층 + 데모 모드

규칙(반드시):
- 페이지/컴포넌트는 `supabase` 를 **직접 import 금지**. `src/lib/data/*` 함수만 호출.
- 각 데이터 함수는 **데모 분기 내장**:
  ```ts
  export async function getWorkspaceWritings(wsId: string) {
    if (isDemoMode()) return MOCK_WRITINGS.filter(w => w.workspace_id === wsId);
    const { data } = await supabase!.from("writings").select("*")...
    return data ?? [];
  }
  ```
- 페이지에서 비동기 로딩은 `useAsync(() => fn(), deps)` → `{ data, loading }`.
- `isDemoMode()`(`src/lib/demo/mode.ts`): 로컬 dev(`DEV`)면 기본 데모,
  `VITE_FORCE_REAL=true` 면 로컬에서도 실 Supabase. 프로덕션은 env 채워지면 실모드.
- 목업 데이터: `src/lib/demo/fixtures.ts` (한국어 카피, ID 네임스페이스 규칙).

## 6. ★ 가시성/공유 모델 (AI office 앱 핵심 참고)

밋업엔 "이 글을 모임에 공유할지, 혼자 볼지"가 **테이블마다 RLS 한 줄**로 구현돼
있다. AI office 의 "워크스페이스 공유 vs 비공개" 기능은 이걸 그대로 베끼면 된다.

### 패턴 A — 워크스페이스 전체 공유 (멤버면 다 봄)

일정·회의록·할일·안건·공지처럼 "모임 공용" 데이터.

```sql
create policy xxx_select on public.xxx
  for select to authenticated
  using ( public.is_workspace_member(workspace_id) );
```

### 패턴 B — 작성자 토글 공유 `is_shared` (★ 가장 많이 씀)

글쓰기·인사이트·프로젝트. 작성자는 본인 것 항상, 남은 **공유(`is_shared=true`)된 것만**.

```sql
create policy writings_select on public.writings
  for select to authenticated using (
    public.is_workspace_member(workspace_id)
    and (is_shared or user_id = auth.uid())   -- ← 공유 OR 내 글
  );
```
- 테이블에 `is_shared boolean default false` 컬럼.
- 작성 폼에 "모임에 공유" 토글 (예: `src/features/projects/project-form.tsx`).
- 목록 쿼리는 RLS 가 알아서 필터 → 프론트는 추가 분기 거의 불필요.

### 패턴 C — 공개 토글 `is_public`, 사용자별 소유

비전보드. 소유자는 항상, 남은 `is_public=true` 일 때만. (`workspace_id` 없이
`user_id` 기준 — 개인 자산형.)

### 패턴 D — 초안/발행 `status: draft|published`

인사이트·글쓰기. `draft` 는 **본인만**, `published` 라야 공유 대상. `is_shared`
와 조합되어 "발행 + 공유" 둘 다여야 남에게 보임. 임시저장은 `/me/posts` 에서 관리.

### 패턴 E — 운영자 모더레이션 (025·039)

작성자 본인 **또는** 워크스페이스 admin/owner 가 UPDATE/DELETE 가능:
```sql
using ( user_id = auth.uid() or public.is_workspace_admin(workspace_id) )
```
운영자는 남의 글을 "숨김/삭제"만, 본문 편집은 UI 에서 안 보이게 함.

### 한눈 표 — 도메인별 가시성

| 도메인 | 테이블 | 가시성 패턴 |
| --- | --- | --- |
| 일정/회의록/일정종류/참석 | meetings, meeting_notes … | A (멤버 전체) |
| 할일 (+좋아요·댓글 047) | tasks, task_likes, task_comments | A |
| 안건·투표 | agendas, agenda_votes, agenda_comments | A |
| 공지 | notices | A (+ 작성 시 푸시) |
| 글쓰기 | writings (+ likes/comments) | B + D |
| 인사이트 | insights (+ likes/comments) | B + D |
| 프로젝트/목표/KPI | projects, goals, kpis | B |
| 독서 챌린지/노트 | readings, reading_notes (+social) | A~B |
| 비전보드 | vision_boards | C (개인·공개토글) |
| 채널 지표(SNS) | channel_* (044–046) | A 조회 / admin 연결 |
| 활동 로그 | activities | A (append-only) |
| 피드백 | dev_messages (043) | 운영자 전용 |

> **AI office 에서 바로 쓸 레시피**: 컨텐츠 테이블에 `workspace_id` +
> `is_shared`(또는 `visibility` enum) 추가 → 패턴 B의 SELECT 정책 →
> 작성 폼에 공유 토글. 멤버 초대는 §3 의 guest→member 흐름 재사용.

## 8. 알림 (푸시) 흐름

```
도메인 변경 (예: createNotice) → notify({ type, workspace_id, actor_id, title, body, url })
   → Edge Function 'notify' (service role)
       1) notification_admin: 운영자 글로벌 토글 확인
       2) 수신자 결정 (전체 멤버 / 운영자 전용 / target_user_ids), actor 제외
       3) notification_prefs: 개인 토글 AND 조합
       4) push_subscriptions 로 웹푸시 발송 (VAPID, _shared/push.ts)
```
- 알림 종류는 `src/lib/push.ts`의 `NOTIFICATION_TYPES` + Edge `EventType` 동기화.
- 새 알림 추가 = 두 곳에 type 추가 + 도메인 함수에서 `notify()` 호출.

## 11. AI office 앱에 적용한다면 (요약)

1. **멤버 초대/가입**: §3 의 `workspace_members` + role(guest→member) + 운영자 수락.
   (초대 코드를 쓰고 싶으면 `workspaces.invite_code` 컬럼 이미 있음.)
2. **공유/비공개 체크**: §6 패턴 B (`is_shared` + RLS `(is_shared or user_id=auth.uid())`).
   "이 글을 워크스페이스에 공유 / 나만 보기" 체크박스가 곧 이 한 컬럼·한 정책.
3. **운영자 모더레이션**: §6 패턴 E.
4. **데이터는 함수 경유 + RLS 최종 방어** + 데모 분기 (§5) 그대로 가져가면 안전.
```

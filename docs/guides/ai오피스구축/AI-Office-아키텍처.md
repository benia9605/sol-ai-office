# AI Office 아키텍처 정리

> Sol의 멀티 브랜드(운명랩 · 쏠닝포인트 · 시목) 운영을 위한 통합 AI Office.
> 이 문서는 "각 앱에 주입 vs 통합 앱" 결정과 그 이유, 권장 데이터 구조, 직원 구성을 정리한 것입니다.
> Claude Code에 JSX와 함께 넘겨서 실제 구현 컨텍스트로 사용하는 용도.

---

## 1. 핵심 결정: 통합 앱 하나 + 각 앱과 연결 (방식 B)

운명랩 · 쏠닝포인트 · 시목 각각의 앱에 AI Office를 심는 대신, **독립된 AI Office 앱 하나**를 만들고 각 브랜드 앱의 데이터를 연결해서 쓴다. 워크스페이스 토글이 곧 "어느 브랜드를 보고 있는가"가 된다.

### 두 방식 비교

**방식 A — 각 앱에 AI Office 주입**
- 운명랩 앱, 쏠닝포인트 앱, 시목 앱 각각에 직원/할일/리포트 기능을 심는다.
- AI Office 코드가 세 벌로 복제된다.
- UI/로직 하나를 고치면 세 군데 다 고치고 세 번 배포해야 한다 → 유지보수 부담 3배.
- Replit 비용을 이미 신경쓰는 상황에서 같은 기능이 세 곳에 박히면 비효율적.

**방식 B — 통합 앱 하나 + 연결 (채택)**
- AI Office는 독립 앱 한 개.
- 브랜드 앱들은 각자 자기 일(리포트 생성, 판매, 쇼핑몰)만 하고, 결과 데이터를 AI Office가 읽어가거나 API/웹훅으로 받는다.
- 워크스페이스 토글 = "어느 앱 데이터를 볼지" 선택.

### B를 택한 이유

1. **AI Office의 본질이 "여러 브랜드를 한 곳에서 본다"는 것.** 워크스페이스 토글 자체가 그 약속이다. A로 가면 브랜드마다 각 앱에 들어가야 해서 토글이 존재할 이유가 사라진다.

2. **데이터가 이미 분리돼 있다.** Sol은 앱마다 Supabase 프로젝트를 따로 두는 SaaS 격리 구조를 쓴다. B는 이 구조와 완벽히 맞는다 — AI Office가 각 Supabase 프로젝트에서 필요한 걸 읽어오면 된다. A는 오히려 이 격리를 깨고 각 앱 안에 AI Office용 테이블을 또 만들어야 한다.

3. **AI 직원 로직은 앱과 독립적이다.** "매일 키워드 수집", "전날 리포트 정리", "SNS 발행" 같은 건 운명랩 앱의 기능이 아니라 그 위에서 도는 자동화다. 앱 안에 넣으면 앱이 무거워지고, 멀티 에이전트(AI 팀) 시스템을 세 앱에 쪼개 넣는 이상한 모양이 된다.

### A가 나은 예외

AI Office 기능을 **고객에게 파는 SaaS 제품**으로 만들 거라면(예: 운명랩 사용자가 자기 운세 앱 안에서 AI 기능을 쓰게 한다면) 각 앱에 임베드하는 게 맞다. 하지만 지금 만드는 건 **Sol 본인(+별이, 민석)이 브랜드를 운영하는 내부 운영 도구**이므로 통합이 정답이다.

## 2. 권장 데이터 연결 구조

두 가지 방식을 섞는 게 현실적이다.

- **집계/참조 데이터** (어제 판매량, KPI 등 자주 안 바뀌는 값)
  → AI Office가 각 브랜드 앱의 Supabase를 **읽기 전용으로 조회**.

- **AI 직원이 만드는 산출물** (키워드 리포트, SNS 초안, 제안, 일일 리포트)
  → AI Office **자체 Supabase 프로젝트에 저장**.

즉 AI Office는 자기 DB를 갖되, 각 브랜드 앱 DB를 "참조"하는 형태. 브랜드 앱들은 AI Office의 존재를 거의 몰라도 된다(낮은 결합도 → 한쪽이 바뀌어도 다른 쪽이 안 깨짐).

```
┌─────────────┐   ┌──────────────┐   ┌──────────┐
│  운명랩 앱   │   │ 쏠닝포인트 앱 │   │ 시목 앱  │
│  Supabase A │   │  Supabase B  │   │ Supabase C│
└──────┬──────┘   └──────┬───────┘   └────┬─────┘
       │ read-only       │ read-only      │ read-only
       └────────────┬────┴────────────────┘
                    ▼
          ┌───────────────────┐
          │    AI Office 앱     │
          │  (자체 Supabase D)  │  ← 직원/일과/일일 리포트/제안 저장
          │                    │
          │  워크스페이스 토글   │  ← 연결된 앱 소스 전환
          └───────────────────┘
```

### 확장성

워크스페이스 = "연결된 앱 소스 설정". 새 브랜드가 생기면 워크스페이스 하나 추가하고 그 앱의 Supabase 연결 정보만 넣으면 끝. (현재 JSX는 워크스페이스가 하드코딩 상태 → 실제 구현 시 설정 테이블로 분리할 것.)

## 3. 화면 구조 (현재 JSX 기준)

좌측 레일 네비게이션으로 화면 전환:

| 메뉴 | 내용 |
|---|---|
| **오늘의 브리핑** | 워크스페이스 전체의 하루치를 한 장으로. 모든 직원 일일 리포트 요약 + 처리해야 할 항목 + 핵심 지표. 매일 아침 이것만 읽으면 어제 전체 파악. |
| **대시보드** | KPI 카드 + 오피스 플로어(직원 현황) + 할일/기록 요약. |
| **AI 직원** | 직원 채용 카드. 누적/오늘/가동 통계. **카드 클릭 → 직원 상세(일일 리포트 아카이브 + 매일 하는 일 + 설정).** |
| **할일** | 칸반(할 일/완료) + 우선순위 + 수동/자동 배지. |
| **일정** | 타임라인, 진행 중 라이브 표시, 담당자. |
| **인사이트** | KPI 큰 차트 + 트렌드 키워드 랭킹 + AI 제안 카드(소구점/마케팅). |
| **기록** | AI/사람 구분 활동 로그. |
| **멤버** | 사람 + AI 직원을 권한과 함께. 이메일 초대. |

### 데이터 성격별 배치 원칙

읽는 흐름이 다른 데이터를 한 곳에 섞지 않는다.

- **매일 전날 리포트(정기 산출물)** → 직원 상세의 "일일 리포트" + 오늘의 브리핑에 집계.
- **트렌드 키워드(시간에 따라 변하는 관찰값)** → 인사이트. 단발 "급상승 감지" 알림은 기록에도 한 줄.
- **CS(처리 상태가 있는 티켓)** → 규모 작으면 인사이트 지표 + 기록 한 줄로 시작, 커지면 별도 CS 메뉴.
- **카테고리 소구점/마케팅(AI 분석·제안)** → 인사이트 하단 "AI 제안" 카드.
- **단발 활동** → 기록.

## 4. AI 직원 구성 (워크스페이스별 현재 배치)

각 직원은 5번의 **타입**을 기반으로 만들어지며, "매일 하는 일(routine)"을 갖고 실행 결과가 날짜별 일일 리포트로 쌓인다. 아래는 워크스페이스별로 현재 배치된 직원과 그 타입.

### 운명랩
- **마케터** (`marketer`) — Threads 콘텐츠 발행, 다음날 초안 작성, 주간 캘린더 정리.
- **SNS 콘텐츠 제작자** (`sns_creator`) — Threads/인스타 게시물을 **실제로 작성**. 자동 발행이 가능하면 발행, 어려우면 **복사용 카피 + 이미지 브리프를 카드로 제공**하고 Sol이 복붙·체크만. (5번의 `sns_queue` UI)
- **리서처** (`researcher`) — 매일 08:00 사주 키워드 검색량 수집, 급상승 감지, 주간 경쟁 스캔.
- **CS봇** (`cs_bot`) — 실시간 문의 1차 응답, 매일 18:00 미해결 요약, FAQ 업데이트 제안.
- **리포트 빌더** (`report_builder`) — 주문 시 리포트 즉시 생성, 야간 배치, 품질 자동 검증.

### 쏠닝포인트
- **콘텐츠 작가** (`marketer`) — 신규 템플릿 소개글, 주간 블로그.
- **SNS 매니저** (`sns_creator`) — 예약 발행 관리, 주간 판매 리포트.

### 시목
- **비주얼 기획** (`sns_creator`) — 인스타 스토리텔링, 신상 콘텐츠 초안(민석 검수).

> 같은 타입이라도 워크스페이스마다 이름·프롬프트·일과가 다르게 설정된다. 타입은 "틀", 프롬프트와 이름이 그 직원의 개성이다.

## 5. 직원 타입 시스템 (핵심 설계)

### 5.1 개념: 직원 = "타입(역할 템플릿)"의 인스턴스

직원은 자유 입력으로 만드는 게 아니라 **미리 정의된 타입을 골라서** 만든다. 타입을 고르면 그 타입에 맞는 **기본 스킬·일과·상세 UI·산출물 형태**가 자동으로 세팅되고, **프롬프트(성격·톤·세부 지시)만 Sol이 직접 입력**한다.

이렇게 하는 이유:
- 타입마다 산출물이 다르므로(SNS 게시물 vs 키워드 리스트 vs CS 티켓) **상세 UI도 타입별로 달라야** 한다. 자유 입력 직원은 어떤 UI를 보여줄지 알 수 없다.
- 채용이 "역할 선택 → 프롬프트 입력" 2단계로 끝나 진입장벽이 낮다.
- 타입별 기본값(일과 주기, 산출물 스키마)이 코드에 정의돼 있어 일관성이 유지된다.

```
새 직원 채용
  → [1] 타입 선택 (카탈로그에서 카드 클릭)
  → [2] 기본 스킬·일과 자동 세팅 (편집 가능)
  → [3] 프롬프트 입력 (이 직원의 성격/톤/세부 규칙) ← Sol이 직접
  → [4] 워크스페이스에 배치 + 가동
```

### 5.2 직원 타입 카탈로그 (이커머스 실전형으로 통합)

Sol이 정리한 6개 커머스 에이전트(SOURCING/DETAIL PAGE/CS/SNS/AD/MONITOR)와 기존 추상 타입을 합쳤다. 커머스 에이전트가 더 구체적이라 이를 코어로 삼고, 운명랩처럼 커머스가 아닌 브랜드에 필요한 타입(리포트 빌더 등)을 보조로 둔다.

각 타입은 `outputKind`(산출물 종류)로 상세 UI가 결정된다. 모든 직원은 **자동 발행이 어려우면 "복붙 가능한 완성본 + 체크" 폴백**을 공통 원칙으로 한다.

#### 코어 — 커머스 운영 (Sol 카탈로그 기반)

| # | 타입 | 역할 한 줄 | 3대 기능 | outputKind (상세 UI) | 난이도 |
|---|---|---|---|---|---|
| 01 | **소싱 기획자** `sourcing` | 팔릴 근거부터 정리 | ① 소싱 가능성 분석(트렌드·경쟁강도 상중하·추천여부·페르소나·채널) ② 키워드 기획안(메인5·서브10·상품명3·태그20) ③ 상품 기획서 초안(타겟·USP·가격·마진·차별화) | `sourcing_brief` — 분석/키워드/기획 탭 + 추천여부 배지 | ⭐⭐ |
| 02 | **상세페이지 제작자** `detail_page` ★최고 임팩트 | 상품정보+사진1장 → 바로 쓰는 상세페이지 | ① 기획안 생성(히어로→문제→솔루션→신뢰→후기→CTA 6섹션) ② 섹션별 카피 재생성(톤 조정) ③ 스마트스토어 HTML 변환 | `detail_builder` — 6섹션 에디터 + 섹션별 재생성 + 이미지 업로드 + 복붙 HTML | ⭐⭐⭐ |
| 03 | **CS 응대** `cs` | 문의 붙여넣기 → 분류+답변초안 | ① 문의 분류+답변(유형/긴급도/초안/추가확인) ② FAQ 10개 생성 ③ 부정리뷰 대응 초안 | `ticket_list` — 티켓 카드(유형·긴급도·답변초안 복사) | ⭐⭐ |
| 04 | **SNS 운영** `sns` | 캘린더+캡션+릴스 스크립트 | ① 월간 캘린더(4주 날짜/유형/주제/해시태그) ② 캡션 A/B/C 3버전 ③ 스토리/릴스 스크립트(15/30/60초 장면별) | `sns_queue` — 캘린더 뷰 + 게시물 카드 큐(복사·발행체크) | ⭐⭐ |
| 05 | **광고 기획** `ad` | 광고 카피·타겟 세팅 가이드 | ① 카피세트(헤드라인5·서브5·상세3·CTA) ② 타겟 세팅(네이버키워드20·인스타 오디언스·일예산) ③ A/B 소재 3세트(감성/기능/가격) | `copy_variants` — 카피세트 + 타겟 가이드 + A/B 비교 | ⭐⭐ |
| 06 | **모니터링** `monitor` | 경쟁사·트렌드 주간 리포트 | ① 경쟁사 분석(가격·리뷰키워드·구성·차별화) ② 트렌드 리포트 ③ 키워드 경쟁강도 예측 | `monitor_digest` — 경쟁사 비교표 + 트렌드 리포트 | ⭐⭐⭐ |

> ⚠️ 06 모니터링: 스마트스토어/쿠팡 공식 API가 제한적이라 **URL/텍스트 붙여넣기 → AI 분석** 방식으로 구현.

#### 보조 — 비커머스·공통 (기존 타입 중 살릴 것)

| 타입 | 역할 | outputKind | 비고 |
|---|---|---|---|
| **리포트 빌더** `report_builder` | 산출물(운세 리포트 등) 생성·검증·배치 | `generation_log` | 운명랩 핵심. 커머스 아님 |
| **리서처** `researcher` | 키워드·트렌드 수집(분석보다 수집 중심) | `keyword_table` | 모니터링과 겹치면 통합 가능 |
| **분석가** `analyst` | KPI 집계·해석·이상치 | `metric_digest` | 데이터 쌓인 뒤 도입 |
| **카피 에디터** `editor` | 교정·톤 통일·브랜드 보이스 검수 | `review_diff` | 다른 직원 산출물 후처리 |

#### 기존 → 통합 매핑 (정리)

- 기존 **마케터** → 04 SNS 운영 + 05 광고 기획으로 분화(역할이 더 명확해짐).
- 기존 **SNS 콘텐츠 제작자** → 04 SNS 운영으로 흡수.
- 기존 **CS봇** → 03 CS 응대 (그대로, 기능 3개로 구체화).
- 기존 **리서처** → 06 모니터링과 역할 중첩 → 수집은 리서처, 분석·예측은 모니터링으로 분리하거나 하나로 합침.
- 기존 **세일즈 라이터** → 02 상세페이지 + 05 광고로 흡수.
- 기존 **커뮤니티 매니저 / 일정 조율가** → 당장 불필요, 후순위 보류.

> 도입 순서 제안: **03 CS → 04 SNS → 01 소싱 → 02 상세페이지(임팩트 최고지만 난이도↑) → 05 광고 → 06 모니터링.** 쉬운 것부터 붙여 운영 흐름을 확인하고, 임팩트 큰 상세페이지는 UX(이미지 업로드+섹션 재생성)에 시간을 더 쓴다.

### 5.2b 24시간 가동 시 API 비용 추산 (일/월)

**전제**
- 모델은 **Sonnet 4.6**($3/1M 입력, $15/1M 출력) 기준. 분류·요약처럼 가벼운 일은 **Haiku 4.5**($1/$5)로 내리면 1/3~1/5로 떨어진다.
- "24시간 알아서"라고 해도 실제로는 **정해진 일과를 하루 몇 번 실행**하는 구조다(상시 추론 아님). 비용은 "실행 횟수 × 1회 토큰"으로 결정된다.
- 1회 실행 토큰은 시스템 프롬프트+컨텍스트(입력)와 산출물(출력)을 합친 대략치. 캐싱/배치 미적용 기준이며, 적용 시 30~50% 추가 절감 가능.
- 환율은 약 1,350원/$로 표기.

| 타입 | 1일 실행 | 1회 토큰(입력/출력) | 1회 비용 | **1일** | **1월(30일)** |
|---|---|---|---|---|---|
| 01 소싱 기획자 | 2회 | 4K / 3K | ~$0.057 | ~$0.11 | **~$3.4 (약 4,600원)** |
| 02 상세페이지 제작자 | 1회(+재생성 2) | 6K / 8K | ~$0.138 | ~$0.30 | **~$9.0 (약 12,200원)** |
| 03 CS 응대 | 20건 | 1.5K / 1K | ~$0.020 | ~$0.39 | **~$11.7 (약 15,800원)** |
| 04 SNS 운영 | 5건 | 3K / 2K | ~$0.039 | ~$0.20 | **~$5.9 (약 7,900원)** |
| 05 광고 기획 | 1회 | 3K / 3K | ~$0.054 | ~$0.05 | **~$1.6 (약 2,200원)** |
| 06 모니터링 | 3회 | 8K / 4K | ~$0.084 | ~$0.25 | **~$7.6 (약 10,200원)** |
| 리포트 빌더 | 30건 | 5K / 6K | ~$0.105 | ~$3.15 | **~$94.5 (약 12.8만원)** |

**해석**
- 리포트 빌더를 뺀 운영 직원 6종을 24시간 돌려도 **월 합계 약 $39 (약 5.3만원)**. 1인 멀티브랜드 운영 도구치고 매우 낮다.
- 비용 폭탄은 거의 항상 **고빈도·고출력 직원**이다. 여기선 리포트 빌더(주문당 긴 리포트 생성)가 단독으로 나머지를 합친 것보다 비싸다 → 이건 "24시간 자동"보다 **주문 발생 시에만 실행**되므로, 실제 비용은 주문량에 비례(위 표는 일 30건 가정).
- **절감 레버:** ① 분류·라우팅·짧은 답변은 Haiku로(3~5배 절감) ② 시스템 프롬프트/상품 정보는 프롬프트 캐싱(캐시 입력 90% 절감) ③ 야간 배치 생성은 Batch API(50% 절감). 셋 다 적용하면 위 추산의 절반 이하로 떨어진다.
- 위 숫자는 **실행 횟수 가정에 전적으로 의존**한다. 실제로는 직원별 일과 주기를 Sol이 설정하므로(5.5의 `Routine.schedule`), 채용 UI에서 "예상 월 비용"을 실행 횟수×모델단가로 미리 보여주면 좋다.

> ⚠️ 가격은 2026년 6월 기준 공개 단가다. 실제 청구 전 [docs.claude.com](https://docs.claude.com)에서 현재 단가를 재확인할 것.



### 5.3 직원 상세 UI는 outputKind에 따라 달라진다

직원 카드를 누르면 뜨는 상세 화면은 **공통 영역 + 타입별 영역**으로 구성된다.

**공통 영역 (모든 타입 동일)**
- 상단: 직원 프로필(이름·역할·상태) + 그 직원의 대표 지표 미니 차트
- 우측: "매일 하는 일(routine)" 목록 + 설정(가동/정지, 실행 시간, 프롬프트 편집)
- 좌측 상단: 일일 리포트 아카이브(날짜별 카드)

**타입별 영역 (outputKind로 분기)**
- `sns_queue` → 캘린더 뷰 + 게시물 카드 큐. 카드마다 본문·해시태그·이미지 브리프 + [본문 복사]/[발행함 체크]/[발행 승인]. (※ 현재 JSX의 SNS 직원이 이 UI)
- `sourcing_brief` → 분석/키워드/기획 탭. 추천여부 배지(추천/보류/비추천) + 경쟁강도 상중하 표시.
- `detail_builder` → 6섹션 에디터(히어로→문제→솔루션→신뢰→후기→CTA). 섹션별 [재생성] + 이미지 업로드 + [스마트스토어 HTML 복사]. **임팩트 최고, UX 난이도 최고.**
- `ticket_list` → 티켓 카드 리스트(유형·긴급도·답변초안 [복사]·추가확인 항목).
- `copy_variants` → 광고 카피세트(헤드라인/서브/CTA) + 타겟 세팅 가이드 + A/B 3세트 비교.
- `monitor_digest` → 경쟁사 비교표(가격·리뷰키워드·구성) + 트렌드 리포트. URL/텍스트 붙여넣기 입력란.
- `generation_log` → 생성 로그 타임라인(건수·소요시간·성공/실패). 리포트 빌더용.
- `keyword_table` / `metric_digest` / `review_diff` → 각 산출물 전용 뷰(보조 타입, 확장 시 구현).

구현 패턴: 상세 화면에서 `switch(staff.type.outputKind)`로 타입별 컴포넌트를 렌더. 공통 레이아웃은 그대로 두고 가운데 산출물 영역만 교체한다.

### 5.4 채용 플로우 UI

"새 직원 채용" 클릭 시:

1. **타입 카탈로그 모달** — 타입 카드 그리드(아이콘·이름·한 줄 설명·outputKind 미리보기 태그). 클릭하면 선택.
2. **세팅 단계** — 선택한 타입의 기본 스킬·일과가 체크리스트로 표시(추가/제거 가능). 실행 주기 선택(실시간/매일/주간).
3. **프롬프트 입력** — "이 직원은 어떻게 일해야 하나" 자유 텍스트. 예시 플레이스홀더 제공(타입별로 다른 예시). 이 부분만 Sol이 직접 작성.
4. **이름 + 워크스페이스 배치** → 생성. 즉시 가동하거나 대기 상태로 둘 수 있음.

현재 JSX의 "빈 자리 · 채용" 카드가 이 플로우의 진입점이 된다.

### 5.5 데이터 형태 (구현 참고)

```ts
// 타입 정의 (코드에 카탈로그로 고정)
type StaffType = {
  key: string;                 // "sourcing" | "detail_page" | "cs" | "sns" | "ad" | "monitor" | ...
  label: string;               // "소싱 기획자"
  defaultSkills: string[];     // 기본 스킬
  defaultRoutines: string[];   // 기본 일과
  features: string[];          // 3대 기능 (모드)
  outputKind: OutputKind;      // 상세 UI 분기 키
  promptPlaceholder: string;   // 프롬프트 입력 예시
  estTokensPerRun: { in: number; out: number };  // 비용 추산용
};

type OutputKind =
  // 코어 (커머스)
  | "sourcing_brief" | "detail_builder" | "ticket_list"
  | "sns_queue" | "copy_variants" | "monitor_digest"
  // 보조
  | "generation_log" | "keyword_table" | "metric_digest" | "review_diff";

// 실제 고용된 직원 (DB 저장)
type Staff = {
  id: string;
  workspaceId: string;
  typeKey: string;             // 어떤 타입인지
  name: string;
  prompt: string;              // Sol이 직접 입력한 부분
  skills: string[];            // 기본값에서 편집된 최종 스킬
  routines: Routine[];
  model: "sonnet" | "haiku";   // 비용/품질 트레이드오프 선택
  state: "working" | "idle";
};

type Routine = { label: string; schedule: "realtime" | "daily" | "weekly"; runAt?: string; runsPerDay?: number };

// SNS 직원의 산출물 예시 (outputKind=sns_queue)
type SnsPost = {
  id: string; staffId: string;
  platform: "threads" | "instagram";
  status: "draft" | "waiting" | "published";  // 초안 / 복붙 대기 / 발행됨
  body: string; hashtags: string[];
  imageNote?: string;          // 이미지/비주얼 브리프
  scheduledFor?: string;
};
```

각 직원의 일일 리포트는 outputKind에 맞춰 요약된다. 예: SNS 직원 = "게시물 N건 작성 / 발행 M건 / 대기 K건", 리서처 = "키워드 N개 수집 / 급상승 K건".

## 6. Claude Code에 넘길 때 메모

- 함께 전달하는 `AIOffice.jsx`는 **구조·디자인 초안**이다. 데이터는 하드코딩된 목업.
- 실제 구현 시 우선순위:
  1. 워크스페이스 → 앱 Supabase 연결 설정 테이블로 분리(하드코딩 제거).
  2. AI Office 자체 Supabase 스키마 설계: `workspaces`, `staff_types`(카탈로그), `staff`, `routines`, `daily_reports`, `suggestions`, `sns_posts`, `todos`.
  3. 직원 타입 카탈로그를 코드 상수로 고정(5.2 표) → 채용 플로우(타입 선택 → 기본 스킬/일과 세팅 → 프롬프트 입력) 구현. 코어 5타입 먼저.
  4. 직원 상세 화면을 `outputKind` 분기 구조로 리팩터(5.3). 공통 레이아웃 + 가운데 산출물 영역만 타입별 교체.
  5. 각 브랜드 앱 Supabase 읽기 전용 조회 레이어(집계 KPI).
  6. AI 직원 실행 엔진: routine을 cron으로 돌려서 daily_report 생성(Claude API + 기존 Teamie AI Office 아키텍처 재활용). 프롬프트는 직원별 `prompt` 필드를 시스템 프롬프트로 주입.
  7. SNS 콘텐츠 제작자(`sns_queue`): 초안 생성 → 복붙/체크 워크플로우 먼저, Threads API 자동 발행은 후순위.
- 스택: React/Vite/TypeScript/Tailwind/Supabase/Claude API, Replit 배포 (기존 스택 유지).
- 디자인 토큰: 크림 배경(#f4f1ea) + 워크스페이스별 액센트 컬러. 자세한 값은 JSX의 `S`/`CSS`/`WORKSPACES` 참고.

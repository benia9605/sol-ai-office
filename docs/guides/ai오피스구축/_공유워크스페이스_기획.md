> ⚠️ **이 문서는 초안 + fable 검토 기록용.** 최종 구현 기준은 **`_공유워크스페이스_확정설계.md`** 참고.

## 1. 핵심 컨셉 — "개인 우선, 공유는 옵트인"

```
지금:  [나] ── 개인 할일/기록/인사이트/AI오피스
이후:  [나] ── 개인 (기본값, 그대로)
         └─ "공유" 토글 ─→ [공유 워크스페이스: 우리부부] ←─ "공유" 토글 ─┘ [남편]
                              · 서로의 공유 항목만 보임
                              · 시목 AI 직원 작업을 같이 봄
                              · 서로에게 할일 추천
```

**3대 원칙**
1. **기본은 전부 private** — 토글 안 켜면 지금과 100% 동일(파트너도 못 봄).
2. **순수 추가형** — 기존 테이블/코드 안 깨짐. 기존 데이터는 자동으로 `private`.
3. **2인으로 시작, N인 확장 가능** — 초대코드 구조는 밋업 스펙 그대로.

## 3. 가시성 모델 (가장 중요한 설계 결정) — ⭐레퍼런스 패턴 B 채택

레퍼런스 §6이 **"공유/비공개 = 테이블마다 `is_shared` 컬럼 + RLS 한 줄"**(패턴 B)로 이미 실증.
그대로 베낀다 → 우리 모든 공유 대상 항목에 **`is_shared boolean default false`** 추가:

| 값 | 의미 | 누가 봄 |
|---|---|---|
| `false` (기본) | 나만 | 작성자 본인만 |
| `true` | 워크스페이스 공유 | 워크스페이스 멤버 전원 |

- 기존 모든 데이터 = `is_shared false` + `workspace_id NULL` 로 백필 → **현 동작 그대로**.
- 작성/편집 모달에 토글 1개: `[ ] 우리 워크스페이스에 공유` → 켜면 `is_shared=true` + `workspace_id` 세팅.
- **할일은 한 단계 더:** 공유 시 `assignee_id`(누구 할일인지) 선택 → "남편에게 추천" = 공유 + assignee=남편.

### 레퍼런스 패턴 → 우리 도메인 매핑
> ⚠️ 핵심 차이: 밋업은 할일·일정이 **패턴 A(멤버 전체 공용)**. 우리는 **기본 개인**이라 할일·일정·기록도 **패턴 B(옵트인 공유)**로 둠.

| 우리 도메인 | 레퍼런스 패턴 | 적용 |
|---|---|---|
| 할일(tasks) | B (+ assignee) | 기본 private, 공유 시 담당자 지정 |
| 기록(journals)·인사이트·독서·스터디 | B | 옵트인 공유 |
| 프로젝트/목표/KPI | B | 시목 = 공유 프로젝트 |
| AI 오피스 리포트(daily_reports 등) | B | 시목 브랜드만 자동 공유 |
| 운영자 수정/삭제 | E | owner가 공유본 숨김/삭제 |

> 💡 2인이라 `is_shared` 이분법으로 충분. N인 확장 시 "특정 멤버만"은 레퍼런스에 없으니 그때 별도 매핑 테이블로. (enum `visibility`로 가도 되지만, **레퍼런스 마이그레이션/폼을 그대로 재사용하려면 `is_shared`가 유리**.)

## 4. 데이터 설계

### 4.1 신규 테이블 (3개만)
```sql
-- 워크스페이스 (부부 = 1개)
workspaces (
  id, name, emoji, color,
  invite_code,        -- 초대 코드
  created_by,
  created_at, updated_at
)

-- 멤버 (다대다)
workspace_members (
  workspace_id, user_id,
  role,               -- 'owner' | 'member'
  nickname,           -- 워크스페이스 내 표시 이름
  joined_at
)

-- (선택) 활동/추천 알림 — 푸시 인프라 이미 있으니 가볍게
workspace_activities (
  id, workspace_id, actor_id,
  action,             -- 'shared_task' | 'recommended_task' | 'shared_insight' | 'shared_record' | 'ai_report_ready'
  resource_type, resource_id,
  metadata,           -- JSONB (제목 등)
  created_at
)
```

### 4.2 기존 테이블 — 컬럼만 추가 (마이그레이션)
> 전부 `nullable` + 기본값으로 안전하게. 기존 행은 자동 `private`/`NULL`.

| 테이블 | 추가 컬럼 |
|---|---|
| `tasks` | `workspace_id`(null), `is_shared`(false), `assignee_id`(null) |
| `journals`(기록) | `workspace_id`, `is_shared` |
| `insights` | `workspace_id`, `is_shared` |
| `schedules` | `workspace_id`, `is_shared` |
| `readings` | `workspace_id`, `is_shared`, `recommended_by` |
| `study_notes` | `workspace_id`, `is_shared` |
| `projects` | `workspace_id`, `is_shared`, `owner_id` ← **시목 = 공유 프로젝트** |

### 4.3 AI 오피스 연동 (24시간 자동 플랜과 합류 지점)
- `daily_reports` / `detail_page_drafts` 에도 `workspace_id` + `is_shared`.
- **시목 브랜드(=shared 프로젝트)의 AI 직원 작업 결과물은 자동으로 `shared`** → 부부가 같은 리포트를 봄.
- 운명랩/쏠닝포인트(개인 브랜드)는 `private` 유지.

## 5. RLS 전략 (개인 보존 + 공유 격리)

밋업 스펙의 헬퍼 그대로:
```sql
CREATE FUNCTION is_workspace_member(ws_id uuid) RETURNS boolean ... -- 멤버 체크
```

각 테이블 SELECT 정책 = **"내 것 OR 내가 속한 워크스페이스의 공유본"** (= 레퍼런스 패턴 B):
```sql
-- 레퍼런스 writings_select 와 동일 구조: (is_shared or user_id=auth.uid())
CREATE POLICY tasks_select ON tasks FOR SELECT USING (
  user_id = auth.uid()                                  -- 내 개인 항목
  OR (is_shared AND is_workspace_member(workspace_id))  -- 공유받은 항목
);

-- 수정: 작성자 또는 (할일이면) 담당자  [+ 패턴 E: owner 모더레이션]
CREATE POLICY tasks_update ON tasks FOR UPDATE USING (
  user_id = auth.uid() OR assignee_id = auth.uid()
);

-- 공유 토글(is_shared/workspace_id 변경)은 작성자만
```
- **핵심:** 기존 동작 보존 — `workspace_id NULL`이면 `user_id = auth.uid()` 한 줄만 타서 지금과 동일.
- 신규 테이블 전부 `user_id` + RLS (기존 17테이블 규칙과 통일).

## 6. UX 설계

### 6.1 워크스페이스 스위처 (헤더/사이드바)
```
[ 👤 내 오피스 ▾ ]   ← 기본
   ├ 👤 내 오피스 (개인)
   └ 🪵 우리 (시목 공동)   ← 선택 시 공유 뷰
```
- **개인 모드:** 지금과 동일(내 private + 내 shared 다 보임).
- **공유 모드:** 워크스페이스 멤버의 `shared` 항목만 모아보기(파트너 것 포함).

### 6.2 각 항목 작성/편집 — 공유 토글 1개
```
┌ 할일 추가 ───────────────┐
│ 제목: [............]      │
│ ...                      │
│ [✓] 우리 워크스페이스에 공유 │
│   └ 담당: ( 나 / 남편 )    │  ← 공유 켜면 등장 (할일만)
└─────────────────────────┘
```

### 6.3 필터 (할일/기록/인사이트 페이지 공통)
`[ 전체 ] [ 내 것 ] [ 공유받은 것 ] [ 내가 추천한 것 ]`

### 6.4 "파트너가 추천한 할일" 인박스
- 남편이 `assignee=나`로 공유 → 내 대시보드에 "추천받은 할일" 섹션 + 푸시.
- 내가 수락 → 내 할일로 편입 / 거절 가능.

### 6.5 AI 직원 같이 보기
- 공유 모드 대시보드에 **"시목 오늘의 오피스 리포트"** 위젯 → 부부 공통.
- 리포트에서 나온 할일을 서로에게 추천/분배.

## 8. 로드맵 (Phase)

- **Phase 1 — 워크스페이스 기반(0.5~1주):** workspaces/members 테이블 + 초대코드 + 스위처 + `useWorkspace`. (아직 공유 항목 없음, 껍데기)
- **Phase 2 — 할일 공유·추천(1주):** tasks에 `visibility/assignee` + 공유토글 + 필터 + "추천받은 할일" + 푸시. (가장 가치 큰 첫 기능)
- **Phase 3 — 기록·인사이트 공유(0.5주):** journals/insights/readings 공유 토글 + 공유 피드.
- **Phase 4 — AI 오피스 합류(24h 플랜과 연결):** 시목 daily_reports 공유 + 부부 공통 대시보드 위젯.
- **Phase 5(선택):** Realtime(공유 항목 즉시 반영), 활동 피드, 통계.

> 24시간 AI 오피스 플랜과 **Phase 4에서 합류** — 둘은 독립적으로 먼저 진행 가능.

# 검토 1 — projects / brand_contexts / workspace 3자 통합

## ① 현 설계 평가

"워크스페이스 = 회사/팀, 항목별 is_shared 토글" 방향 자체는 맞습니다. 문제는 질문에 이미 드러나 있습니다 — **시목이 workspace인지 project인지 헷갈린다는 것 자체가, 두 개념이 다른 축이라는 신호**입니다.

## ② 문제점

1. **워크스페이스와 브랜드는 다른 축입니다.** 워크스페이스의 본질은 *사람의 묶음(멤버십·권한 경계)*이고, 브랜드(=project)의 본질은 *사업 단위(AI 컨텍스트·파이프라인 실행 단위)*입니다. "시목 = workspace"로 만들면 이 둘이 융합돼서, ⓐ 부부가 나중에 시목 외 두 번째 공동 프로젝트(예: 함께 하는 새 브랜드, 가계부, 이사 준비)를 시작하는 순간 워크스페이스를 또 만들고 멤버십을 중복 관리해야 하고, ⓑ "운명랩을 나중에 공유로 전환"하려면 항목 이동이 아니라 *워크스페이스 신설 + 전체 데이터 이전*이 됩니다.
2. **projects가 4역할(브랜드컨텍스트+공유단위+AI소유자+워크스페이스 소속)을 겸하면 안 됩니다.** 특히 "공유 단위" 역할이 위험합니다 — 전제에서 이미 공유는 *항목별 is_shared*로 정했으므로, project에 또 공유 플래그를 두면 가시성 판정이 2중 규칙(프로젝트 공유 AND 항목 공유?)이 돼서 RLS와 UX 양쪽이 복잡해집니다.
3. 이전 검수의 "brand_contexts → projects FK"는 유지해야 합니다. project ≠ 항상 브랜드이기 때문입니다(예: "이사 준비", "딴길청년들" 같은 비브랜드 프로젝트). brand_contexts를 projects에 흡수하면 비브랜드 프로젝트마다 NULL 컬럼 무더기가 생깁니다.

## ③ 개선안 — 역할을 4계층으로 분리

```
workspaces        = 사람 묶음 (권한 경계)      ← "누가 보나"
projects          = 사업/활동 단위              ← "무슨 일이냐" (workspace_id FK)
brand_contexts    = AI 파이프라인 설정 (1:1)    ← "AI가 뭘 아냐" (project_id FK, 브랜드인 project만)
items (tasks 등)  = 콘텐츠 (workspace_id + is_shared + user_id)
```

**시목에 대한 답: 시목은 project입니다.** 부부 팀 워크스페이스(이름은 "시목"이라 지어도 무방하지만, *모델상으로는* 멤버 {Sol, 민석}인 team workspace)를 하나 만들고, 그 안에 project 시목을 둡니다. 운명랩·쏠닝포인트는 Sol의 개인 워크스페이스 안의 project. 이렇게 하면 "운명랩을 나중에 공유"는 `UPDATE projects SET workspace_id = ...` 한 줄(+ 하위 항목 일괄 이동)이 됩니다.

권장 스키마:

```sql
create table workspaces (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  type text not null check (type in ('personal','team')) default 'team',
  created_by uuid not null references auth.users(id),
  created_at timestamptz default now()
);

create table workspace_members (
  workspace_id uuid references workspaces(id) on delete cascade,
  user_id uuid references auth.users(id) on delete cascade,
  role text not null check (role in ('owner','member')) default 'member',
  joined_at timestamptz default now(),
  primary key (workspace_id, user_id)
);

-- 기존 projects에 추가
alter table projects add column workspace_id uuid references workspaces(id);

-- brand_contexts는 이전 검수안 유지 (project_id unique FK)
-- 콘텐츠 테이블 공통 추가분
alter table tasks
  add column workspace_id uuid references workspaces(id),
  add column is_shared boolean not null default true,
  add column assignee_id uuid references auth.users(id);
-- schedules / insights / journals 도 workspace_id + is_shared 동일 추가
```

파이프라인 연결: `daily_reports`·`agent_runs`는 `project_id`(어느 브랜드의 작업인지)를 갖고, 가시성은 그 project의 workspace를 따릅니다. 시목 데일리 리포트는 자동으로 부부 둘 다에게 보이고 **푸시도 멤버 전원에게** 가야 합니다(기존 push.ts는 단일 user 가정 — 수정 대상).

# 검토 2 — 멀티 워크스페이스 + 통합 뷰

## ① 현 설계 평가

"흩어져서 정신없어진다"는 우려는 정확하고, 통합 뷰를 1급 요구사항으로 둔 것도 옳습니다. 다만 통합 뷰를 "특별한 기능"으로 설계하려는 접근이 함정입니다 — **RLS를 제대로 짜면 통합 뷰는 '워크스페이스 필터를 안 거는 평범한 쿼리'일 뿐**입니다. 그게 이 설계의 성공 기준입니다.

## ② 문제점 (질문에 없는 구멍 포함)

1. **개인 공간을 `workspace_id NULL`로 두면 17개 테이블 전부에서 RLS·뷰·쿼리에 NULL 특례가 박힙니다.** `(workspace_id is null and user_id = auth.uid()) or (...)` 식의 분기가 모든 정책에 복제되고, 한 테이블에서만 빠뜨려도 누수 또는 안 보임 버그입니다. NOT NULL 제약도 못 겁니다.
2. **Postgres VIEW + RLS 함정:** 밋업식 UNION 뷰를 그대로 확장하면 됩니다만, Supabase에서 뷰는 기본적으로 **뷰 소유자 권한으로 실행**됩니다 → 기반 테이블 RLS가 *무시*됩니다. `security_invoker = true` 없이 만들면 통합 캘린더가 남의 비공개 항목까지 노출하는 보안 사고가 납니다. 밋업의 calendar_events가 단일 사용자 전제로 만들어졌다면 이 옵션이 빠져 있을 가능성이 높으니 확인 필수.
3. **가장 큰 구멍 — 질문에 없는 것: AI 컨텍스트 주입 경계.** 현재 `buildSystemPrompt`는 *그 유저의 모든* 프로젝트·KPI·할일·요약을 주입합니다. 민석님이 시목 워크스페이스에서 모디에게 질문하면, RLS는 통과 못 해도 **Sol의 운명랩 KPI가 시스템 프롬프트를 타고 답변에 새어 나올 수 있습니다.** Edge Function은 service role이라 RLS를 우회하므로, 컨텍스트 조합 쿼리를 "현재 워크스페이스 + is_shared(또는 본인 소유)" 기준으로 직접 스코핑해야 합니다. RLS만 믿으면 안 되는 유일한 경로이고, 공유 기능에서 제일 먼저 터질 곳입니다.
4. **"할일 추천 = 즉시 상대 할일 생성"의 두 가지 누수:** ⓐ `assignee_id = 상대`인데 `is_shared = false`면 상대가 자기 할일을 못 보는 모순 → RLS에 "assignee는 항상 본다" 절이 필요합니다. ⓑ 이전 검수에서 지적한 할일 목록 오염이 이제 *남의 목록*을 오염시킵니다. 즉시 생성은 두 분이 정한 방향이니 유지하되, 최소한 assignee에게 푸시 + "받은 할일" 구분 표시(category나 `created_by ≠ assignee_id`로 필터)는 넣으세요.
5. **journals(일기)에 공유 default를 적용하면 사고 납니다.** 아침/저녁 일기가 팀 워크스페이스에 shared-default로 들어가는 순간 프라이버시 사고입니다. 가시성 default는 테이블 일괄이 아니라 **타입별**이어야 합니다.

## ③ 개선안

### (a) 개인 공간 = '개인 워크스페이스 행' 방식 (NULL 방식 기각)

가입(및 마이그레이션) 시 유저당 `type='personal'` 워크스페이스 1개 + 본인 멤버십을 자동 생성합니다. 모든 콘텐츠 행은 `workspace_id NOT NULL`. 효과: RLS 정책이 17개 테이블에서 **단 하나의 동일한 패턴**이 되고, 개인↔공유 이동이 `workspace_id` UPDATE 하나가 되며, 통합 뷰에 COALESCE 분기가 사라집니다. UI에서 개인 공간을 워크스페이스 목록과 구분하고 싶으면 `type='personal'` 필터로 처리(공유 토글도 personal에선 숨김 — 무의미하므로).

### (b) RLS — 멤버십 함수 + 단일 패턴

```sql
-- 멤버십 조회 (STABLE: 쿼리당 1회 평가, 플래너 캐싱)
create or replace function my_workspace_ids()
returns setof uuid language sql stable
security definer set search_path = public as $$
  select workspace_id from workspace_members where user_id = auth.uid()
$$;

-- 모든 콘텐츠 테이블 공통 SELECT 패턴 (tasks 예시)
create policy tasks_select on tasks for select using (
  user_id = auth.uid()                                      -- 내 것은 항상
  or assignee_id = auth.uid()                               -- 배정받은 것은 항상
  or (is_shared and workspace_id in (select my_workspace_ids()))
);

create policy tasks_insert on tasks for insert with check (
  user_id = auth.uid()
  and workspace_id in (select my_workspace_ids())
);

create policy tasks_update on tasks for update using (
  user_id = auth.uid()
  or assignee_id = auth.uid()                               -- 상태 변경용
  or (is_shared and workspace_id in (select my_workspace_ids()))  -- 부부 공동 편집 허용
);

create policy tasks_delete on tasks for delete using (user_id = auth.uid());
```

성능: 솔직히 2인·수천 행 규모에선 무엇을 해도 빠릅니다. 다만 위 함수 패턴 + 인덱스 `(workspace_id, is_shared)`, `(user_id)`, `(assignee_id)`만 깔아두면 멤버 수가 늘어도(딴길청년들을 워크스페이스로 들일 가능성) 그대로 갑니다. 과설계는 여기까지면 충분합니다.

### (c) 통합 뷰 — UNION VIEW 확장 (security_invoker 필수)

```sql
create or replace view unified_agenda
with (security_invoker = true)   -- ★ 이게 없으면 RLS 우회 = 보안사고
as
select 'schedule' as kind, id, workspace_id, user_id, is_shared,
       title, date as on_date, time as on_time, null::text as status
from schedules
union all
select 'task', id, workspace_id, user_id, is_shared,
       title, due_date, null, status
from tasks where due_date is not null;
```

기반 테이블 RLS가 invoker 권한으로 그대로 적용되므로 **뷰 자체엔 권한 로직이 0줄**입니다. 그래서:
- 통합 뷰 = `select * from unified_agenda where on_date between ...` (워크스페이스 필터 없음)
- 워크스페이스 뷰 = 같은 쿼리 + `and workspace_id = :current`

"정신없음" 해결은 데이터가 아니라 표현 계층입니다: 통합 화면에서 **워크스페이스별 컬러 도트 + 워크스페이스 토글 칩(다중 선택)**. 기존 `Layout.tsx`가 상태 허브이므로 `currentWorkspaceId`(null = 통합)를 여기에 두고 모든 훅이 구독하면 구조 변경이 최소화됩니다. 밋업 calendar_events 패턴은 이 형태로 멀티 워크스페이스 확장에 문제없습니다 — 단 invoker 옵션과 `workspace_id` 컬럼 통과, 이 두 가지만 챙기면 됩니다.

### (d) 기본 가시성 — "팀 워크스페이스는 공유 default, 단 타입별 예외"

권장: **team 워크스페이스에서 tasks/schedules/insights는 `is_shared=true` default, journals(일기·회고)는 `false` default.**

근거: 항목을 시목 워크스페이스에 *넣는 행위 자체가 이미 공유 의사 표현*입니다. 비공개 default면 부부의 가장 빈번한 동선(시목 일정 잡기)마다 토글 한 번이 추가되고, "분명 넣었는데 왜 상대한테 안 보여?"가 일상 버그 신고가 됩니다. 잘못 공유(부부 사이) < 잘못 비공개(협업 마비)입니다. 반대로 일기는 잘못 공유가 훨씬 치명적이므로 역방향 default. 멘탈 모델은 한 문장으로: **"어디(워크스페이스)에 넣었는지가 공유를 정하고, is_shared는 예외적 숨김 스위치다."**

### (e) 마이그레이션 — 기존 17테이블 무중단 순서

핵심 원리 두 가지: *모든 단계가 가산적(additive)일 것*, 그리고 *Postgres RLS 정책은 OR로 합산되므로 기존 정책을 지우지 말고 새 정책을 얹을 것*.

1. `workspaces`·`workspace_members` 생성 + RLS.
2. 기존 유저 전원에 personal 워크스페이스 + 멤버십 백필. 신규 가입용 트리거(`auth.users` insert → personal ws 자동 생성)도 이때.
3. 콘텐츠 테이블에 `workspace_id`(NULLABLE로 시작)·`is_shared`·(tasks만) `assignee_id` 추가 — 앱은 아직 아무것도 모름, 무영향.
4. 백필: 전 행 `workspace_id = 본인 personal ws`. 행 수 검증.
5. `NOT NULL` + FK + 인덱스 확정.
6. 새 SELECT/UPDATE 정책 추가(기존 `user_id = auth.uid()` 정책은 그대로 둠 — OR 합산이라 기존 동작 100% 보존).
7. 프론트: services에 workspace 컨텍스트 추가하되 default = personal → **워크스페이스 스위처를 출시하기 전까지 앱 동작이 완전히 동일**한지 이 시점에서 확인. ★CLAUDE.md 룰: `types.ts` / `*.service.ts` / `mockSupabase.ts` 3곳 동시 수정 — mockSupabase에는 워크스페이스 필터링 시뮬레이션까지 넣어야 로컬에서 공유 시나리오를 테스트할 수 있습니다.
8. 스위처 + 시목 team 워크스페이스 생성 + 민석님 초대(이메일 초대 플로우가 필요 — `workspace_members`에 pending invite 처리, 기획서에 없으면 추가 필요) + `projects.workspace_id` 이동(시목 1건) + 시목 하위 기존 항목 일괄 이동 여부는 **사용자 선택**으로(과거 데이터까지 자동 공유하면 놀랄 수 있음).
9. 마지막으로 Edge Functions(데일리 파이프라인·모디 컨텍스트)의 워크스페이스 스코핑(②-3) — 이걸 8보다 먼저 끝내지 않으면 민석님 초대 순간이 곧 누수 시작점입니다.

## 한 장 요약

1. **시목 = project. 워크스페이스는 사람 묶음(부부 팀), 브랜드는 그 안의 프로젝트.** projects의 4역할 겸직 반대 — brand_contexts 분리 유지.
2. **개인 공간은 NULL이 아니라 personal 워크스페이스 행** — RLS가 17테이블에서 단일 패턴이 됩니다.
3. **통합 뷰는 기능이 아니라 "필터의 부재"** — RLS를 제대로 깔면 공짜. 단 VIEW에 `security_invoker = true` 누락 시 보안사고.
4. **최대 구멍은 RLS가 아니라 AI 컨텍스트 주입** — service role 경로의 buildSystemPrompt를 워크스페이스 스코프로 막기 전엔 민석님을 초대하지 마세요.
5. **default는 팀=공유, 일기만 비공개** + assignee는 is_shared와 무관하게 항상 본인 할일을 보는 RLS 절 추가.

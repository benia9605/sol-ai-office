# AI 직원별 전용 뷰 · 레이아웃 설계 검증 요청서

> **목적:** Sol AI Office의 "회사 오피스" — AI 직원들이 일하고, 그 결과(일일 리포트)를 **직원마다 완전히 다른 전용 화면**으로 보여줘야 한다.
> 지금은 일부 직원만 전용 뷰가 있고, 그나마 출력 스키마(`OUTPUT_SCHEMA`)가 축약돼 있어 GPT가 SOP에서 설계한 풍부한 구조가 다 안 담긴다.
> **이 문서를 검토하고: ① 직원별 출력 스키마를 어디까지 풀버전으로 끌어올릴지 ② 각 직원의 전용 뷰 레이아웃(서로 다른 메뉴/형식)을 어떻게 잡을지** 검증·보완해 주세요.

---

## 0. 대전제 (꼭 지켜야 할 설계 원칙)

1. **직원마다 화면이 다르다.** 같은 카드 반복 ❌. 소싱은 "의사결정 브리프", CS는 "티켓 인박스", 분석가는 "지표 대시보드"처럼 **UI 패러다임 자체가 직원별로 다름**.
2. **출력은 2층 구조.** ①마크다운 본문(body, 사람이 읽는 리포트) ②`content_json`(전용 뷰가 그리는 구조화 데이터). 전용 뷰는 ②를 그린다.
3. **데이터가 없으면 안 그린다.** 필드가 비면 해당 블록 숨김(옵셔널 렌더).
4. **HITL(사람 승인) 일관.** AI가 만든 일정/할일/발송은 자동 등록 ❌ → "승인 대기" 큐로.
5. **공통 푸터:** 모든 리포트엔 코멘트(내 의견) 기능 + 자동/수동 배지 + 날짜.

## 1. 전체 현황 한눈에

| # | 직원 | typeKey | outputKind | 전용 뷰 | 현재 상태 | 제안 레이아웃 형식 |
|---|------|---------|-----------|---------|-----------|-------------------|
| ① | 소싱 기획자 | `sourcing` | `sourcing_brief` | SourcingView | ✅ 구현(축약) | **의사결정 브리프** (판정 히어로+5축+마진) |
| ② | 상세페이지 제작자 | `detail_page` | `detail_builder` | — | ⬜ 미구현 · SOP 초안 작성됨(`09_상세페이지.md`, 검증 대기) | **페이지 빌더** (6섹션 스테퍼) |
| ③ | CS 응대 | `cs` | `ticket_list` | TicketListView | ✅ 구현(축약) | **티켓 인박스** (요약바+티켓 큐) |
| ④ | SNS 운영 | `sns` | `sns_queue` | SnsQueueView | ✅ 구현(축약) | **콘텐츠 캘린더/큐** (믹스 도넛+카드) |
| ⑤ | 광고 기획 | `ad` | `copy_variants` | CopyVariantsView | ✅ 구현(축약) | **A/B 3세트 비교** (나란히 카드) |
| ⑥ | 모니터링 | `monitor` | `monitor_digest` | — | ⬜ 미구현 | **경쟁사 대시보드** (비교표+알림 피드) |
| ⑦ | 분석가 | `analyst` | `metric_digest` | — | ⬜ 미구현 | **KPI 대시보드** (지표 카드+추이+이상치) |
| ⑧ | 비주얼 디렉터 | `visual` | `image_brief` | — | ⬜ 미구현 | **촬영/프롬프트 보드** (컷리스트+프롬프트+후보 갤러리) |
| ⑨ | 운영 매니저 | `ops` | `ops_digest` | — | ⬜ 미구현 | **일일 지휘 콘솔** (Top3+승인큐+직원 health) |

## 2. 직원별 상세 (현재 ↔ SOP 풀버전 ↔ 제안 레이아웃)

각 직원: **(A) 현재 스키마** → **(B) 현재 전용 뷰** → **(C) SOP 풀버전에서 빠진 필드** → **(D) 제안 전용 레이아웃** → **(E) 검증 질문**

---

### ① 소싱 기획자 — `sourcing_brief` · "의사결정 브리프"

**(A) 현재 스키마**
```json
{"verdict":"추천|보류|비추천","score":"N/10","confidence":"높음|중간|낮음","reasons":[],
 "scores":{"trend":0,"entry":0,"target":0,"margin":0,"diff":0},
 "margin":{"price":0,"cost":0,"netRate":""},
 "persona":"","channels":[],"risks":[],"nextActions":[],"sources":[]}
```

**(B) 현재 전용 뷰 (SourcingView)** — 판정 배지+점수+신뢰도 / 판정이유 3줄 / 5축 게이지(●●○ 2점 만점) / 마진 카드(판매·원가·순마진율) / 페르소나 / 채널 칩 / 리스크 칩 / 다음액션 체크박스 / 출처

**(C) SOP 풀버전에서 빠진 필드**
- `margin` 세부: 플랫폼수수료·포장배송·예상광고비(매출15%)·**순마진 금액**·**손익분기 월 판매량** (지금은 price/cost/netRate 3개뿐)
- `scores` 축별 **근거(reason)** — 지금은 숫자만, "왜 트렌드 상인지" 없음
- `sources` 구조화 `{title,url,type}` (출처종류=공식몰/쇼핑몰/블로그)
- `nextActions` 구조화 `{type,title,priority}` (할일 자동등록 연결용)
- `handoff`(다른 직원 넘기기): 상세페이지/광고/SNS/비주얼/모니터링별 넘길 내용
- 모드별: 키워드 모드(메인5/서브10/상품명3/태그20+경쟁강도), 기획서 모드(USP3/가격포지셔닝/구매장벽+극복법)

**(D) 제안 레이아웃**
```
┌ 판정 히어로 ─────────────────────┐
│ [추천 ✅]  8/10  · 신뢰도 높음     │   ← 큰 배지 + 점수 게이지
│ 안 팔릴 이유부터 3줄                │
├ 5축 레이더 ──────┬ 손익 분기점 ───┤   ← 좌:5축(레이더/바, 축별 근거 툴팁)
│ 트렌드 ●●○ 근거… │ 판매39,000      │     우:마진 워터폴(판매-원가-수수료
│ 마진   ●●○ 근거… │ -원가-수수료-광고 │        -배송-광고=순마진)
│                  │ =순마진 OO(54%)  │     +손익분기 월 40개 강조
│                  │ 손익분기 월 40개 │
├ 타겟 페르소나 카드 ───────────────┤
├ 채널·리스크 칩 ──────────────────┤
├ 다음 검증 액션 [할일 등록] ────────┤   ← 체크 → HITL 할일 큐
├ 넘기기: 상세📄 광고🎯 SNS📣 비주얼📸 │   ← 핸드오프 칩(클릭 시 해당 직원에게)
└ 출처 (url+신뢰도) ───────────────┘
[모드 탭] 분석 / 키워드 / 기획서       ← 모드별로 본문 구조 전환
```

**(E) 검증 질문**
1. 마진을 **워터폴(폭포) 차트**로 보여주는 게 과한가, 표가 나은가?
2. 5축은 **레이더 차트** vs **가로 바**? 모바일 가독성 기준 추천?
3. 키워드/기획서 모드는 같은 화면 **탭 전환** vs 별도 리포트?

### ② 상세페이지 제작자 — `detail_builder` · "페이지 빌더" (⚠️ SOP 문서 없음)

**(A) 현재 스키마**
```json
{"brief":{"target":"","usp":[],"banned":[]},
 "sections":[{"key":"","title":"","coreLine":"","bullets":[],"visual":"","cta":""}],
 "complianceCheck":{"pass":true,"flags":[]}}
```

**(B) 현재 전용 뷰** — **없음** (마크다운 본문만)

**(C) 갭** — SOP 문서 자체가 없음. 코드만 존재. → **이 직원은 SOP부터 새로 써야 할지** 판단 필요.

**(D) 제안 레이아웃**
```
┌ 상단 brief: 타겟 · USP칩 · 금지표현 ─┐
├ 6섹션 세로 스테퍼 ─────────────────┤
│ ①히어로  핵심카피 / 비주얼지시 / [재생성]│  ← 섹션마다 카피+이미지 브리프
│ ②문제제기 …                          │     +섹션별 재생성 버튼
│ ③솔루션(기능→이익) …                 │
│ ④신뢰/증거 …                         │
│ ⑤후기 …                              │
│ ⑥CTA …                               │
├ 컴플라이언스: 과장/효능 플래그 ───────┤
└ [전체 HTML 복사] [스마트스토어용 변환] ┘
```

**(E) 검증 질문**
1. 상세페이지 직원 **SOP를 새로 작성**할까요? (다른 8명은 GPT 검증 SOP 있음)
2. 6섹션은 **세로 스테퍼** vs **아코디언**? 섹션별 재생성 UX 어떻게?

### ③ CS 응대 — `ticket_list` · "티켓 인박스"

**(A) 현재 스키마**
```json
{"tickets":[{"type":"","urgency":"즉시|일반|낮음","sentiment":"긍정|중립|불만|격앙",
  "draft":"","needsConfirm":"","needsHumanApproval":false}]}
```

**(B) 현재 전용 뷰 (TicketListView)** — 티켓 카드 리스트: 긴급도 이모지(🔴🟡🟢)+유형+긴급도+감정 칩+승인배지 / 답변초안 박스 / 확인필요 / 복사

**(C) SOP 풀버전에서 빠진 필드**
- 상단 **summary 헤더**: `{total, urgent, needs_human_approval, faq_candidates, negative_reviews}` — 인박스 요약바
- 티켓 레벨: `risk_level`(이탈위험), `approval_reason`(승인 사유), `faq_candidate{should_create,question,answer}`, `next_actions`(상세페이지 개선 등 넘기기), `status`(신규/초안/승인대기/완료/보류)
- `needsConfirm` 배열화(여러 확인 항목)

**(D) 제안 레이아웃**
```
┌ 요약바: 총12 · 긴급3 · 승인대기2 · FAQ후보4 · 부정후기1 ┐  ← 인박스 헤더
├ [긴급순 정렬] 티켓 큐 ──────────────────────────────┤
│ 🔴 격앙 · 환불 · 이탈위험高  [승인 후 발송]           │  ← 감정/위험 색상
│    답변초안 …………………… [복사] [승인]              │     승인대기는 강조
│ 🟡 중립 · 배송문의                                    │
│    답변초안 … · FAQ후보 ⭐(자동 FAQ 등록 제안)        │
├ 부정후기 대응 (별도 섹션) ──────────────────────────┤
└ 넘기기: 상세페이지 개선 인사이트 N건 ────────────────┘
```

**(E) 검증 질문**
1. **인박스(메일함) 메타포**가 맞나? 아니면 카드 그리드?
2. 승인 대기 티켓을 **상단 고정** vs 인라인 강조?

### ④ SNS 운영 — `sns_queue` · "콘텐츠 캘린더/큐"

**(A) 현재 스키마**
```json
{"posts":[{"date":"","channel":"","format":"","objective":"","hook":{"type":"","text":"","score":0},
  "body":"","hashtags":{"large":[],"medium":[],"small":[],"brand":[]},
  "imageBrief":"","variants":["A","B","C"],"status":"draft"}]}
```

**(B) 현재 전용 뷰 (SnsQueueView)** — 포스트 카드: 날짜+채널+포맷+목적+상태 / 훅(점수) / 본문 / 해시태그 그룹 / 이미지브리프 / variants 칩 / 복사

**(C) SOP 풀버전에서 빠진 필드**
- 상단 **summary**: `{total, needs_approval, scheduled, needs_visual, mix:{정보팁,공감일상,상품,후기UGC,비하인드}, warnings:["판매글 과다"]}` — **콘텐츠 믹스 비율 경고**(SOP 최강조)
- post: `time`, `content_type`(유형), `cta:{type,text}`(현재 CTA 칸 자체 없음), `next_actions`, `id`
- `variants` 객체화 `{label,angle,hook,body}` (지금은 "A","B" 문자열)

**(D) 제안 레이아웃**
```
┌ 콘텐츠 믹스 도넛 ────────────────┐
│ 정보30 공감25 상품20 후기15 비하인드10 │  ← 유형 비율 시각화
│ ⚠️ 이번 주 판매글 과다(40%)         │     + 경고 배너
├ [월간 캘린더] or [카드 큐] 토글 ────┤  ← 보기 전환
│ 6/17 인스타·피드·저장               │
│   훅: "도마 하나 바꿨을 뿐인데" 8/10  │  ← 훅 점수 배지
│   본문 … CTA … #해시태그            │
│   [A][B][C] 캡션 탭 · 이미지브리프🖼  │  ← A/B/C 탭 전환
│   [복사] [발행 승인]                 │
└ 넘기기: 비주얼(이미지 N건 필요) ──────┘
```

**(E) 검증 질문**
1. 기본 보기는 **캘린더** vs **카드 큐**? (자동 일과는 매일 1건, 캘린더가 빌 수 있음)
2. 콘텐츠 믹스 경고를 얼마나 강하게? (도넛 vs 단순 바)

### ⑤ 광고 기획 — `copy_variants` · "A/B 3세트 비교"

**(A) 현재 스키마**
```json
{"sets":[{"type":"감성|기능|가격","headline":"","sub":"","detail":"","cta":"","status":"테스트"}],
 "targeting":{"keywords":[],"audience":{},"budgetSplit":{}},"complianceCheck":{"pass":true}}
```

**(B) 현재 전용 뷰 (CopyVariantsView)** — 3세트 grid 카드(유형/헤드라인/서브/상세/CTA/상태+복사) / 타겟팅 카드(키워드·코어·예산) / 컴플라이언스 한 줄

**(C) SOP 풀버전에서 빠진 필드 (⚠️ BASE_SOP는 출력하라는데 스키마에 칸 없음)**
- 세트별: `imageDirection`(이미지 방향), `recommendedChannels`(적합 채널), `expectedMetric`(기대 지표), `caution`(주의점), **`variant_id`**(분석가 성과 연동)
- `product{name,price,channels,goal,angle}`
- `audience{core,lookalike,interest,exclude}` (지금 빈 객체)
- `channelGuide{instagram,naver,kakao}` 채널별 가이드
- `complianceCheck` 항목별: `{exaggeration,unverifiedNo1,lowestPrice,healthClaim,landingConsistencyRequired}`
- `handoff{sns,analyst,landing,ownerApproval}`

**(D) 제안 레이아웃**
```
┌ 상품/목표 바: 티크도마 · 전환 · 콜드 ──────────────┐
├ 3세트 나란히 비교 ─────────────────────────────────┤
│ [감성형]      [기능형]      [가격형]                │
│ 헤드라인       헤드라인      헤드라인                │
│ 서브·상세      서브·상세     서브·상세               │
│ CTA            CTA           CTA                     │
│ 🖼이미지방향   🖼            🖼                       │
│ 📍적합:인스타  📍네이버      📍카카오                │
│ 📊기대:저장↑   📊클릭↑       📊전환↑                 │
│ ⚠️주의…        status:테스트  variant_id:AD-03       │
│ [복사]         [복사]        [복사]                  │
├ 타겟팅 패널: 키워드20 · 코어/유사/관심/제외 · 예산배분 ┤
├ 컴플라이언스 체크리스트 ────────────────────────────┤
│ ☑과장 ☑1위표현 ☑최저가 ☑효능 ☐랜딩일관성(확인필요)  │
└ 넘기기: SNS · 분석가(성과추적) · 랜딩 · 사장승인 ─────┘
```

# GPT 답변
쏠닝님, 지금 방향은 맞습니다.
그리고 솔직히 말하면, 지금 막힌 지점도 너무 자연스러워요.

SOP는 이미 충분히 잘 잡혀 있는데, 문제는 그 풍부한 내용을 화면이 못 받아먹고 있는 상태예요. 즉 지금은 AI 직원은 똑똑해졌는데, 직원 전용 책상이 너무 작아서 결과물이 다 안 펼쳐지는 상황입니다.

결론부터 말하면 이렇게 가야 합니다.

스키마는 전 직원 풀버전으로 확장하되, UI는 MVP 필수 블록부터 렌더링한다.
즉 content_json은 넉넉하게 받고, 화면은 데이터가 있는 블록만 보여주는 방식이 맞습니다. 그래야 나중에 직원이 성장해도 스키마를 계속 갈아엎지 않습니다.

아래는 검증/보완안입니다. 그대로 개발 문서에 붙여 넣어도 되게 정리했습니다.

AI 직원별 전용 뷰 · 레이아웃 설계 검증 보완안

목적: Sol AI Office의 AI 직원들이 각자 다른 방식으로 일하고, 각 직원의 결과물이 전용 화면에서 풍부하게 보이도록 content_json 스키마와 전용 뷰 레이아웃을 재설계한다.
핵심 결론: 모든 직원이 같은 카드 UI를 쓰면 안 된다. 단, 완전히 제각각 만들면 유지보수가 어려우므로 공통 셸 + 직원별 본문 패러다임으로 설계한다.

⸻

1. 최종 결론

현재 설계 방향은 맞다.

AI 직원별 전용 뷰는 반드시 달라야 한다.
소싱은 의사결정 브리프, CS는 티켓 인박스, SNS는 콘텐츠 큐, 광고는 A/B 비교, 분석가는 KPI 대시보드, 운영매니저는 지휘 콘솔이 되어야 한다.

다만 구현 전략은 아래처럼 나누는 것이 가장 안정적이다.

공통 골격은 통일한다.
본문 레이아웃은 직원별로 완전히 다르게 한다.
스키마는 풀버전으로 확장한다.
렌더링은 MVP 필수 블록부터 한다.
비어 있는 필드는 숨긴다.
승인/발송/집행은 모두 HITL 승인 큐로 보낸다.

⸻

2. 지금 문제의 본질

현재 문제는 SOP가 부족한 것이 아니라, SOP의 풍부한 출력 구조를 담을 그릇이 작다는 것이다.

현재 구조는 대략 아래 상태다.

BASE_SOP:
직원별 역할, 판단 기준, 넘기기, 컴플라이언스까지 풍부함
OUTPUT_SCHEMA:
핵심 몇 줄만 받는 축약형
전용 뷰:
4명만 구현
구현된 4명도 축약 스키마 기준
결과:
AI는 많이 말할 수 있는데,
content_json은 적게 담고,
전용 뷰는 더 적게 보여준다.

따라서 해야 할 일은 3가지다.

1. outputKind별 content_json 스키마 확장
2. 직원별 전용 뷰 9종 구현
3. 공통 HITL/댓글/날짜/배지/핸드오프 구조 통일

⸻

3. 전체 설계 원칙

3-1. 2층 출력 구조 유지

모든 직원 리포트는 반드시 2층 구조로 간다.

body:
사람이 읽는 마크다운 리포트
content_json:
전용 뷰가 렌더링하는 구조화 데이터

역할을 분리해야 한다.

영역	목적
body	자연어 보고서, 사장님이 읽는 설명
content_json	전용 뷰, 카드, 표, 큐, 버튼, 승인 흐름
metadata	자동/수동, 날짜, 직원, outputKind, 상태

3-2. 전용 뷰는 content_json 우선

전용 뷰는 마크다운을 파싱해서 그리면 안 된다.
반드시 content_json을 기준으로 그린다.

좋은 구조:
AI → body + content_json 생성
전용 뷰 → content_json만 보고 렌더링
나쁜 구조:
AI → 마크다운 생성
전용 뷰 → 마크다운 문자열에서 억지로 파싱

3-3. 비어 있으면 숨긴다

모든 블록은 옵셔널 렌더링한다.

{data.handoff?.length > 0 && <HandoffPanel />}
{data.approvalQueue && <ApprovalQueue />}
{data.sources?.length > 0 && <SourceList />}

필드가 없으면 빈 박스가 보이면 안 된다.
AI 직원이 아직 충분히 데이터를 못 만들었어도 화면은 깔끔해야 한다.

3-4. 스키마는 넉넉하게, UI는 단계적으로

스키마를 MVP 수준으로만 만들면 나중에 다시 확장할 때 깨진다.
반대로 UI를 한 번에 풀버전으로 구현하면 개발 부담이 크다.

따라서 이렇게 간다.

스키마:
풀버전 지향
UI:
MVP 블록 우선 렌더링
확장:
필드가 들어오면 추가 블록이 자동 표시

⸻

4. 공통 레이아웃 셸

직원별 화면은 달라야 하지만, 전체 리포트의 바깥 골격은 통일한다.

공통 셸 구조

┌──────────────────────────────────────────────┐
│ 직원 리포트 헤더                               │
│ 직원명 · outputKind · 자동/수동 · 생성일 · 상태 │
├──────────────────────────────────────────────┤
│ 직원별 전용 본문                               │
│ sourcing / cs / sns / ad / monitor / analyst...│
├──────────────────────────────────────────────┤
│ 공통 푸터                                     │
│ 댓글 · 내 의견 · 승인 상태 · 원문 보기          │
└──────────────────────────────────────────────┘

  "staffName": "광고 기획자",
  "typeKey": "ad",
  "outputKind": "copy_variants",
  "runMode": "auto",
  "createdAt": "2026-06-16T08:30:00+09:00",
  "status": "draft",
  "needsApproval": true

Q1. 마진은 워터폴 차트가 과한가?

데스크톱에서는 워터폴 추천, 모바일에서는 표/스택 카드 추천.

소싱에서 마진은 핵심 판단 요소다.
단순히 “순마진율 35%”만 보여주면 사장이 어디서 비용이 빠지는지 모른다.

다만 모바일에서 워터폴은 복잡할 수 있다.

권장:

데스크톱:
마진 워터폴
모바일:
판매가
- 원가
- 수수료
- 포장배송
- 예상광고비
= 순마진

Q2. 5축은 레이더 차트 vs 가로 바?

모바일 기준 가로 바 추천.

레이더 차트는 예뻐 보이지만 모바일에서 숫자와 근거를 읽기 어렵다.
소싱은 감상용 차트가 아니라 판단용이므로 가로 바가 낫다.

권장:

진입장벽    ●○○  근거 보기
타겟적합    ●●○  근거 보기
마진        ●●○  근거 보기
차별화      ●○○  근거 보기

수동으로 “광고 카피만 만들어줘”라고 하면
targeting, budgetSplit이 없을 수 있음.
이때 타겟팅 패널은 숨김.

20. 최종 결론

Sol AI Office가 실제로 회사처럼 굴러가려면, 직원별 화면은 단순 리포트 카드가 아니라 직무별 업무 공간이어야 한다.

소싱은 상품을 들여올지 결정하는 책상
상세페이지는 페이지를 조립하는 작업대
CS는 고객 문의를 처리하는 인박스
SNS는 콘텐츠 발행 큐
광고는 A/B 테스트 비교판
모니터링은 경쟁사 상황판
분석가는 KPI 관제실
비주얼은 촬영/프롬프트 보드
운영매니저는 오늘의 지휘 콘솔

가장 중요한 것은 운영매니저다.
운영매니저가 모든 직원의 제안, 승인, 실패, 누락을 묶어 오늘 볼 것 3개로 올려야 사장이 실제로 움직일 수 있다.

따라서 개발 우선순위는 아래를 추천한다.

1. content_json 스키마 v2 확장
2. 운영매니저 전용 뷰 구현
3. 분석가 전용 뷰 구현
4. 광고/SNS/CS 기존 뷰 고도화
5. 모니터링/비주얼/상세페이지 신규 뷰 구현
6. 소싱 뷰 마진/근거/핸드오프 고도화

이 순서로 가면 AI 직원들이 단순히 “말하는 존재”가 아니라, 실제로 회사 안에서 제안하고, 넘기고, 승인받고, 실행을 추적하는 조직처럼 보이기 시작한다.

쏠닝님에게 제일 필요한 건 “더 많은 리포트”가 아니라 사장님이 매일 결정을 덜 하게 만드는 구조예요.
그래서 저는 운영매니저를 제일 먼저 제대로 세우는 걸 추천합니다.

지금 설계에서 가장 중요한 3개만 뽑으면 이거예요.

1. content_json을 풀버전 지향으로 확장하되, UI는 MVP 필드만 먼저 렌더링
2. 모든 직원의 공통 셸은 통일하고, 본문 패러다임은 완전히 다르게
3. 운영매니저가 전 직원의 승인 대기·누락·실패를 모아 “오늘 볼 것 3개”로 압축

이 구조가 잡히면 AI 직원들이 단순히 글을 생성하는 게 아니라, 정말로 회사 안에서 각자 역할을 맡아 움직이는 느낌이 납니다.
